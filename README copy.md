# BitsKeep

BitsKeep は電子部品の在庫管理と、設計業務で使う案件参照をまとめて扱う Web アプリケーションです。

## できること

- 部品管理
  - 部品一覧、部品詳細、部品登録/編集、部品比較
  - 分類、パッケージ、スペック、商社、在庫情報を横断して管理
- 在庫運用
  - 棚一覧
  - 在庫警告一覧
  - 入庫/出庫履歴
- 案件管理
  - 案件一覧、案件詳細、案件別部品構成
  - Notion の `01_案件管理` DB との同期
- 設定/管理
  - Notion 連携設定
  - ホーム設定（主要アクション並び替え）
  - マスタ管理、ユーザー管理、監査ログ、CSVインポート
- 設計支援ツール
  - エンジニア電卓
  - 合成抵抗/容量ネットワーク探索
  - 設計ツールハブ

## 主要画面

認証後の主な画面は以下です。

- `/dashboard`
  - トップ画面。グローバル検索、主要アクション、最近使った情報への入口
- `/components`
  - 部品一覧。探索画面として使い、詳細確認、比較、編集へつなぐ
- `/components/create`
  - 部品登録画面。新規登録と既存部品編集を兼用
- `/components/{id}`
  - 部品詳細。スペック、仕入先、在庫、案件利用、履歴の確認
- `/projects`
  - 案件一覧。Notion 同期結果と独自案件を統合表示
- `/stock-alert`
  - 在庫警告一覧。逼迫部品の確認と対処起点
- `/locations`
  - 棚管理。棚ごとの保管状況を確認・更新
- `/settings/integrations`
  - Notion 連携設定
- `/settings/home`
  - ホーム設定。主要アクションの表示/順序設定
- `/tools/calc`
  - エンジニア電卓
- `/tools/network`
  - 合成抵抗/容量ネットワーク探索
- `/tools/design`
  - 設計ツールハブ

## API 概要

API は `bits-keep/routes/api.php` で定義しており、基本的に `auth` 必須です。主な責務は以下です。

- マスタ管理
  - `GET/POST/PUT/DELETE /api/categories`
  - `GET/POST/PUT/DELETE /api/packages`
  - `GET/POST/PUT/DELETE /api/spec-types`
- 部品管理
  - `GET/POST/PUT/DELETE /api/components`
  - `PATCH /api/components/{component}/{section}`
  - `GET /api/components/compare`
  - `GET /api/components/{component}/similar`
  - `GET /api/components/{component}/transactions`
  - `POST /api/components/{component}/stock-in`
  - `POST /api/components/{component}/stock-out`
- 在庫/棚/商社
  - `GET/POST/PUT/DELETE /api/locations`
  - `POST /api/locations/inventory`
  - `GET /api/stock-alerts`
  - `GET/POST/PUT/DELETE /api/suppliers`
- 案件管理
  - `GET /api/projects/options`
  - `GET /api/project-businesses`
  - `GET /api/projects/sync/status`
  - `POST /api/projects/sync/notion`
  - `GET /api/projects/sync-runs`
  - `GET/POST/PUT/DELETE /api/projects`
  - `GET/POST/PATCH/DELETE /api/projects/{project}/components...`
- 設定
  - `GET/PUT /api/settings/integrations/notion`
  - `GET/PUT/DELETE /api/preferences/{key}`
- 管理機能
  - `GET /api/users`
  - `POST /api/users/invite`
  - `PATCH /api/users/{user}/role`
  - `PATCH /api/users/{user}/active`
  - `GET /api/audit-logs`
- 計算
  - `POST /api/calc/networks/search`

詳細な入出力は `詳細仕様/仕様書.md` を正本とします。

## 案件名参照仕様

2026-04-07 時点で、BitsKeep が扱う案件名は以下を統合します。

- Notion 配下の `010_xxx`〜`099_xxx` 事業ページにある `01_案件管理` DB
  - 現時点の例: `010_綾電クリエイト / 01_案件管理`
  - 現時点の例: `011_eleXlink / 01_案件管理`
- BitsKeep 内で独自に登録した案件

Notion 側ではタイトル項目 `案件` を案件名として扱います。同期対象は DB 名が `01_案件管理` で、親ページ名が `010_`〜`099_` に一致する事業のみです。`998_事業テンプレート` のようなテンプレート系は除外します。

案件選択UIはコンボボックス方式です。入力中に事業名・案件名・案件番号で候補をリアルタイム絞り込みし、既存案件は候補から選択します。一致候補がない場合のみ新規追加候補として扱い、確認モーダルを経て独自案件を作成します。

案件管理画面の Notion同期UI は、長い説明文を読む前提にしません。`未設定 / 要確認 / 未実行 / 実行中 / 0件 / 成功 / 失敗` の状態を先に見せ、不足項目と次アクションを状態カードで判断できる構成を基本方針とします。

## 認証方針

- ログインIDは当面 `メールアドレス` を維持する
- 表示名は `users.name` を使う
- `メールアドレス` はログイン、パスワード再設定、招待、将来の通知・確認メールに使う
- SNSログインは将来拡張として検討し、初期候補は `Google` と `GitHub` とする
- SNSログインを導入しても、既存のメール+パスワード認証は残し、`verified email` を軸に安全にアカウント連携する

## 計算系アドオン

- エンジニア向け電卓
  - 進数混在、SI接頭辞、履歴再計算、ユーザー定義定数/関数、数値解法を扱える高機能電卓を追加予定
- 合成抵抗/容量ネットワーク探索
  - 単純な直列/並列だけでなく、複数素子の複雑ネットワーク探索と分圧計算まで扱えるツールへ拡張予定
- 設計解析ツールハブ
  - ADCコード/スケーリング
  - 電解コンデンサ寿命推定
  - 誤差/歩留まり解析
  - サーミスタ分圧・温度変換
  - シャント抵抗/電流検出誤差
  - I2C/UART余裕解析
  - 電源余裕解析
  - 比較器しきい値/ヒステリシス解析
  - ロジックレベル適合チェッカ
  - ロジックICリファレンス
  - 周波数応答/ボード線図解析

専門家4視点で相談した結果、初期実装の最優先帯は `誤差/歩留まり解析` `電源余裕解析` `ADCコード/スケーリング設計` `インタフェース余裕解析` `電流検出解析` とする。採用済みの他ツールも維持するが、先行実装はこの5本から着手する。

設計解析ツールは、`最悪/標準/最良` `温度・ばらつき` `pass/fail と margin` `支配要因` `コード化係数または部品候補` のどれを返すかが明確であることを共通条件とする。単純な増幅率計算や単純なCRカットオフ計算は、より高次の解析表示が付く場合のみ再検討する。

専門家相談の詳細Q&Aは `制作管理/専門家相談記録/2026-04-08_計算系アドオン専門家相談.md` に保存する。要約だけでなく、質問文と各専門家の回答を残す。

将来候補として、`熱見積もり` `部品ディレーティング確認` `電池寿命/消費電力見積もり` `MOSFET損失/熱/SOA解析` `インダクタ飽和/リップル/コア損失解析` `I/Oタイミング予算解析` `簡易伝送線路/終端/反射見積もり` を仕様書に記録する。まだ実装チェックリストには載せない。

別枠の斬新候補として、`起動・停止/リセット/依存関係診断` `立ち上げ/量産デバッグ/観測性診断` `アナログスイッチ/多重化前段成立性` `入力保護/注入電流/保護協調診断` `代替部品差分評価` `FPGAピン/バンク整合` `RTL/ファーム/レジスタ整合` `ノイズ侵入経路/リターンパス評価` を追加記録した。`入力保護/注入電流/保護協調診断` は、ヒューズ、TVS、ポリスイッチ、eFuse、逆接保護まで含める前提で扱う。詳細Q&Aは `制作管理/専門家相談記録/2026-04-08_斬新設計判断ツール専門家相談.md` に保存する。初回は記録止まりだったが、後段の実現可否整理で一部を実装対象へ引き上げている。

反証許可付き再インタビューでは、`起動・停止/リセット/依存関係診断` と `入力保護/注入電流/保護協調診断` の優先度をもっと上げるべき、`周波数応答/ボード線図解析` と `ロジックICリファレンス` は主役ではなく補助扱いに寄せるべき、という指摘が重なった。詳細Q&Aは `制作管理/専門家相談記録/2026-04-08_反証許可付き追加インタビュー.md` に保存する。

このうち `起動・停止/リセット/依存関係診断` は、詳細ドラフトを先行具体化したうえで、テンプレート限定の初版実装対象へ引き上げた。入力項目、内部モデル、判定ルール、暫定API、UIモックを `詳細仕様/仕様書.md` と `詳細仕様/UI構成/startup_sequence_diag.html` に落とし、実装進捗チェックリストへ反映する。

さらに、非計算系だが設計ミス低減に効く候補として `通信系コネクタピン配/視点補助リファレンス` を追加記録した。オス/メス、mating face/solder side、前面/背面の取り違えを減らすための参照ツールである。

高頻度・小修正系に軸を切り替えた相談では、`コネクタ視点補助リファレンス` `制約付き抵抗ネットワーク探索` `DIP/ジャンパ/ストラップ設定逆引き` `ピン1・極性・向き・シルク照合` が強く挙がった。見落とされていたのは `毎回の小さい視点変換・記法変換・参照ミス` であり、詳細Q&Aは `制作管理/専門家相談記録/2026-04-08_高頻度小修正系ツール専門家相談.md` に保存する。

熱設計系では、`抵抗` `Reg` `Tr` を対象に、熱抵抗チェーンの各段で温度上昇を積み上げ表示する `熱設計 / 熱抵抗チェーン可視化` を詳細ドラフト化する。各熱抵抗の寄与が見える段積みグラフを主役にする。

## 全体詳細設計と実現可否

全体構成は、`Core Inventory` `Project Master` `Search & Compare` `Tool Hub` `Calc Foundation` `Search/Optimize Engines` `Rule/Graph Diagnostics` `Reference Catalog` `External Sync & Audit` `User Preference` の10モジュールに分けて考える。計算・診断ロジックの正本はサーバ側に置き、フロントは入力補助と可視化に徹する。

実現可否は次の4段で扱う。

- `初版でそのまま実装可`
- `スコープ限定で実装可`
- `追加調査/データ整備後に実装可`
- `現時点見送り`

このうち `初版でそのまま実装可` と `スコープ限定で実装可` は実装進捗チェックリストへ落とし込み、`追加調査/データ整備後に実装可` と `現時点見送り` は仕様書に `現時点では実現困難` として残す。

この整理では、`高頻度・小修正系` と `高損害診断系` を分けて評価する。前者は 5 秒で使える軽量参照・小ツールを重視し、後者は入力負荷が増えても `次に直す1項目` を返せる診断力を優先する。

初版の投入順は、`運用基盤` → `計算系初版本隊` → `高頻度・小修正系` → `高損害診断系` とする。`起動依存診断` と `熱抵抗チェーン可視化` はテンプレート限定で先行可能、`入力保護/保護協調` は部品モデルと基本形テンプレートを揃えてから入れる。

## トップ画面の考え方

BitsKeep のトップ画面は、全画面への単純なリンク集ではなく、作業開始地点として設計します。

- 最上段は部品・案件・棚・機能を横断するグローバル検索
- 次段は頻出操作だけに絞ったクイックアクション
  - クイックアクションカードはユーザー自身が並べ替え可能（ドラッグ&ドロップ、設定はサーバー保存）
  - 在庫警告バッジは固定表示
- その下に在庫警告・登録案件・最近更新の「今日の確認事項」カード
- 全機能一覧セクションを最下部に配置し、連携設定・ユーザー管理もここから入る
- ヘッダとナビは目次リンクのみ。説明文は置かない

UI設計基本方針: **説明文を読ませない。状態と操作ラベルで自明にする。**

## 案件番号表示

Notion同期で取得した案件は `external_code_案件名`（例: `RC260000100_イルインソールV0.1.1開発`）形式で表示します。
`external_code` は Notion の `案件番号` プロパティ（formula型対応）から取得します。
