### 2-1. マスタ管理（SCR-009）
- [x] 分類 CRUD API + Vue画面
- [x] パッケージ CRUD API + Vue画面
- [x] スペック種別 CRUD API + Vue画面（単位一覧含む）
- [x] マスタ削除方針を `完全削除` `アーカイブ(論理削除)` `復元` の3段へ再設計
  - 対象: `分類` `パッケージ` `スペック種別` `商社` `保管棚`
  - なぜだめか: 現状は「削除」という1語で、実際には `参照中なら拒否` `在庫がなければ削除可` `NULL化して削除可` など挙動がバラバラである。ユーザーは削除結果を予測できず、運用上は「使わないが履歴は残したい」ケースを扱えない。
  - どう直すか: 標準操作を `アーカイブ` に変更し、通常一覧と選択候補からは隠すが既存参照は保持する。`復元` を管理者操作として必須実装し、`完全削除` は未使用・影響なしの場合だけ別操作で許可する。
- [x] 分類/パッケージ/スペック種別へ論理削除と復元を実装
  - なぜだめか: これらは部品に紐づいたまま生き続けるマスタであり、今の「使用中は削除拒否」だけでは、運用上は不要になった候補を現役一覧から外せない。逆に物理削除すると過去の意味づけが壊れる。
  - どう直すか: 各マスタへ `SoftDeletes` を導入し、通常一覧・部品登録コンボボックス・部品一覧フィルタ候補からは除外する。管理者向けに `アーカイブ済みを表示` と `復元` を用意し、既存部品詳細では `アーカイブ済み` と分かる表示を足す。
- [x] マスタ名の一意制約を、アーカイブ/復元運用を含めて再定義
  - なぜだめか: 論理削除を入れると、`同名の新規作成` と `旧データの復元` が衝突する。ここを決めないまま `deleted_at` だけ足すと、復元時に失敗するか、同名マスタが併存して意味が壊れる。
  - どう直すか: 原則として「アーカイブ中も名前は予約する」を採用し、復元は元名で戻せるようにする。名前の再利用は `完全削除` 後のみ許可し、バリデーションと一意インデックスをそれに合わせて組み直す。
- [x] マスタ一覧の危険操作UIを `削除` 主体から `アーカイブ` 主体へ変更
  - なぜだめか: 現在の一覧は `編集` と同じ密度で `削除` が並んでおり、何が起きるかも見えない。論理削除前提なら、危険操作の主語は `消す` ではなく `現役候補から外す` であるべきで、今のラベルは誤解を招く。
  - どう直すか: 一覧の行操作は `編集` `アーカイブ` `完全削除(条件成立時のみ)` へ再編し、`使用件数` `削除可否` `アーカイブ済み` を同じ行で見せる。確認ダイアログには影響件数と復元可否を明記する。
- [x] 分類/パッケージ/スペック種別の新規追加エラーを修正
  - なぜだめか: `StoreCategoryRequest` `StorePackageRequest` `StoreSpecTypeRequest` の `unique:...,'.$id` が PostgreSQL で `id <> ''` を生成し、新規追加が SQL エラーで落ちる。マスタ追加の基本操作が成立していない。
  - どう直すか: 3 Request すべてを `Rule::unique()->ignore($id)` へ置換し、新規作成時に空文字がSQLへ流れないよう統一する。重複時はDBエラーではなく通常のバリデーションエラーへ戻す。
- [x] 並び順を持つのに並び替え操作がない問題を解消
  - 対象: `分類` `パッケージ` `スペック種別`
  - なぜだめか: UI は `sort_order` を見せているのに、実際の運用では数字を手入力して保存するしかない。並び指定機能を持つと言いながら、操作系が実装されていない。
  - どう直すか: 一覧上で上下移動またはドラッグ並び替えを実装し、保存時に `sort_order` を一括更新する。少なくとも「順序を見ながら動かせる」UI を用意する。
- [x] スペック種別の単位設計を `複数候補` 前提から見直す
  - なぜだめか: 現状は `spec_units` テーブルと UI が「単位候補を複数入力する」前提で組まれている。ユーザー要件は「単位欄は最初から1つ見える」「複数入力は不可」「0個にもできる」であり、今の仕様と逆向きである。
  - どう直すか: `StoreSpecTypeRequest` `SpecTypeController` `master-list` UI を、`単位は0または1件` のモデルへ再設計する。複数単位候補テーブルを残すか単一カラムへ寄せるかを判断し、少なくとも UI では1入力欄のみを標準表示する。

### 2-3. 部品登録/編集（SCR-003）

- [x] 分類/パッケージ/商社のインライン新規追加で PostgreSQL エラーが出る問題を修正
  - なぜだめか: 新規作成時の `unique` バリデーションが `id <> ''` という不正SQLを生成し、UI上は「その場で追加できる」と見えているのに保存以前で破綻する。部品登録の連続入力が止まり、コンボボックス化の価値を壊している。
  - どう直すか: `StoreCategoryRequest` `StorePackageRequest` `StoreSupplierRequest` の `unique:...,'.$id` をやめ、`Rule::unique(...)->ignore($id)` へ統一する。新規時は空文字をSQLへ流さず、重複時だけ正常にバリデーションエラーを返す。
- [x] 部品登録画面の「商社を新規追加」UIと権限設計を一致させる
  - なぜだめか: 画面では誰でも商社を新規追加できるように見えるが、サーバ側は `Supplier` 作成を `admin` 限定にしている。権限不足ユーザーには達成不能な操作を見せており、UIと権限制御が矛盾している。
  - どう直すか: `admin` 以外には商社の新規追加候補を出さず既存選択だけにする。もし編集者にも許可するなら要件を明文化したうえで `StoreSupplierRequest::authorize()` を見直す。
- [x] 部品登録画面の下部アクション行を追加
  - なぜだめか: 保存ボタンがヘッダ上部にしかなく、長いフォームの入力完了後に毎回上まで戻る必要がある。フォーム長と操作位置が釣り合っておらず、入力作業を無駄に往復させている。
  - どう直すか: フォーム最下部に `登録/更新` と `戻る` を置き、上部と同じ主操作を複製する。長大フォームでは開始点と終了点の両方に保存導線を持たせる。
- [x] 部品登録/編集に `代表保管棚` の単一選択を追加
  - なぜだめか: 現状の部品登録フォームには棚項目がなく、登録時点で「この部品は普段どこに置くか」を決められない。保管棚は在庫ブロック上の実棚と、部品マスタ上の定位置を分けて扱うべきだが、その後者が欠けている。
  - どう直すか: `components` に `primary_location_id` などの代表保管棚カラムを追加し、部品登録/編集/詳細に `代表保管棚` を単一選択で持たせる。これは実在庫数量を持つ棚ではなく、日常参照用の定位置として扱い、入庫時の初期値にも使う。
- [x] 部品登録画面の未保存離脱ガードを `saving` ベースから `dirty` ベースへ変更
  - なぜだめか: 現状の確認は「保存処理中」しか見ておらず、実際に危険なのは「入力済みだが未保存」の離脱である。ブラウザ戻るやパンくずで警告なく入力が消えるのは、フォーム画面として致命的。
  - どう直すか: 初期値との差分で `dirty` を持ち、未保存変更がある間は `beforeunload` と画面内リンク遷移で確認を出す。保存成功時だけ `dirty=false` に戻す。
- [x] 部品画像の保存方式を、ファイル名規則つきサーバ保存へ統一
  - なぜだめか: 現状は UUID 命名で `components/images` へ保存しており、人が見ても何のファイルか分からない。要件は「DBにはファイル名、実体はサーバフォルダ、名前は `型番_通称_分類` ベース」である。
  - どう直すか: `FileStorage::storeComponentImage()` を差し替え、ファイル名を `型番_通称_分類` 正規化 + 重複時 `_01`〜`_99` で採番する。DB には相対パスまたはファイル名のみ保持する方針を明文化する。
- [x] データシート保存方式を画像と同じ命名規則へ統一し、複数登録に対応
  - なぜだめか: 現状は `components.datasheet_path` の単一カラムしかなく、1部品1PDFしか持てない。要件は複数データシート対応であり、現行スキーマでは表現不能である。
  - どう直すか: `component_datasheets` のような別テーブルを追加し、`file_path` `original_name` `sort_order` `note` を持たせる。既存 `datasheet_path` からの移行方針を決め、登録UIも複数添付前提へ作り直す。
- [x] パッケージ違い派生部品の複製登録フローを設計・実装
  - なぜだめか: 同一部品の DIP/SOP などを「1部品に複数パッケージ」で表すと、価格やスペック差分を潰してしまう。一方、毎回ゼロから再入力させると派生登録コストが高すぎる。
  - どう直すか: `部品複製` 機能を追加し、基本情報・分類・スペック・仕入先・画像・PDF をコピーした新規登録へ入れるようにする。画像/PDF は未変更なら元ファイルを参照し、差し替え時のみ新しい実体へ切り替える。元部品との関連を `derived_from_component_id` 等で残すか検討する。
