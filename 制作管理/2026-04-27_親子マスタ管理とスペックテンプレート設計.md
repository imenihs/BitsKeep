# 親子マスタ管理とスペックテンプレート設計メモ

作成日: 2026-04-27

## 背景

秋月電子通商の実売部品を参考にマスタを増やした結果、パッケージとスペック項目は実用的になった一方で、件数増加により次の問題が出た。

- マスタ管理で全件を横断して並び替えると、親分類をまたいだ順序管理になり運用が重い。
- 部品登録ではスペック項目の候補が多すぎ、`select` 形式では目的の項目を選びにくい。
- `TRならこのパラメータ`、`OPアンプならこのパラメータ`、`マイコンならこのパラメータ` という実務上の定型があるのに、現状は毎回手でスペック行を足す必要がある。

## 決定

パッケージは既存の `パッケージ分類 -> パッケージ` を正式な親子管理単位とし、並び替えも分類内に限定する。

スペックは画面上は `スペック分類 -> スペック項目` の親子として扱う。ただし `電源電圧`、`動作温度`、`端子数` のような共通スペックは複数分類に出るため、DBは単純な `spec_types.group_id` ではなく `spec_groups` と `spec_group_spec_type` の多対多にする。

さらに、部品登録の初期行生成には `スペックテンプレート` を使う。分類グループは候補を絞るための束、テンプレートは実際に入力行を作るための束として分ける。

## データモデル方針

### パッケージ

既存構造を維持する。

```text
package_groups
  -> packages
components.package_id
```

- `packages.sort_order` は `package_group_id` 内の表示順として扱う。
- 全件横断の並び替えはやめる。
- 一括並び替えAPIも分類内だけを更新する。

### スペック分類

```text
spec_groups
  id
  name
  description
  sort_order
  deleted_at

spec_group_spec_type
  id
  spec_group_id
  spec_type_id
  sort_order
  is_required
  is_recommended
  default_profile
  default_unit
  note
```

- `spec_types` はスペック項目の正本として維持する。
- `spec_types` に `group_id` は直接持たせない。
- グループ内表示順、必須/推奨、既定profile、既定単位は pivot 側に持たせる。
- `共通` グループを用意し、他グループの候補にも混ぜられるようにする。

### スペックテンプレート

```text
spec_templates
  id
  spec_group_id
  name
  description
  sort_order
  deleted_at

spec_template_items
  id
  spec_template_id
  spec_type_id
  sort_order
  default_profile
  default_unit
  is_required
  note
```

- テンプレート適用時に `component_specs` の入力行を生成する。
- テンプレートは部品カテゴリや解析結果から推奨できる。
- 同じスペック項目が複数テンプレートに入ってよい。

### カテゴリ連動

```text
category_spec_group
  category_id
  spec_group_id
  sort_order
  is_primary
```

- 部品登録で分類を選んだとき、対応するスペック分類とテンプレートを上位候補に出す。
- 複数カテゴリ選択時は `is_primary` と `sort_order` を優先し、重複するスペック項目は1回だけ表示する。

| OPアンプ/コンパレータ | 回路数、電源電圧、入力オフセット電圧、入力バイアス電流、GBW、スルーレート |
| ロジックIC | 電源電圧、入力数、出力数、伝播遅延時間、最大発振周波数、出力電流 |

## 初期テンプレート案

- `BJT基本`: VCEO、コレクタ電流、hFE、fT、全損失
- `MOSFET基本`: VDSS、ドレイン電流、オン抵抗、ゲートしきい値電圧、ゲート電荷、全損失
- `ダイオード基本`: ピーク耐圧、平均順電流、順方向電圧、逆回復時間
- `LED基本`: 順方向電圧、順方向電流、発光波長、光度、指向角
- `LDO基本`: 入力電圧、出力電圧、出力電流、ドロップアウト電圧、消費電流
- `OPアンプ基本`: 回路数、電源電圧、入力オフセット電圧、入力バイアス電流、利得帯域幅積、スルーレート
- `ロジックIC基本`: 電源電圧、入力数、出力数、伝播遅延時間、出力電流
- `マイコン基本`: 電源電圧、フラッシュ容量、RAM容量、GPIO数、ADCチャンネル数、クロック周波数
- `センサ基本`: 電源電圧、消費電流、測定範囲、分解能、通信系の自由属性
- `発振子基本`: クロック周波数、負荷容量、周波数許容差、温度周波数特性

通信方式、I2Cアドレス、ロジックファミリなどの文字列寄り項目は、数値比較軸として使う予定がなければスペック項目ではなくカスタムフィールドに逃がす。

## UI方針

### マスタ管理

- パッケージタブは、左に `パッケージ分類`、右に分類内の `パッケージ` を表示する。
- パッケージの並び替えは右ペインの分類内だけで行う。
- スペック管理は `スペック分類` と `スペック項目` を分ける。
- 左に `スペック分類`、右にその分類へ所属するスペック項目を表示する。
- 右ペインでは `必須/推奨/任意`、既定profile、既定単位、並び順を編集できる。
- スペック項目自体の編集と、分類内での表示順編集を分ける。
- `未分類スペック` ビューを用意し、どの分類にも所属していない詳細を見つけられるようにする。

### 部品登録

- スペック項目の巨大 `select` は廃止する。
- 部品カテゴリ選択後、推奨される `スペック分類` と `テンプレート` を表示する。
- テンプレート適用でスペック入力行を生成する。
- スペック行の項目選択は `現在のスペック分類内候補 + 共通候補 + 検索結果` に絞る。
- 例外時だけ `全スペック項目から検索` を開く。
- 解析ヘルパーは抽出した `part_kind` やカテゴリ候補からテンプレートを推定し、レビュー画面で適用候補として出す。

## API方針

- `GET /api/package-groups/{id}/packages` または既存 `GET /api/packages?package_group_id=` を分類内管理の正規導線にする。
- `PUT /api/package-groups/{id}/packages/reorder` で分類内の `packages.sort_order` を一括更新する。
- `GET /api/spec-groups?with_spec_types=1`
- `POST /api/spec-groups`
- `PUT /api/spec-groups/{id}`
- `DELETE /api/spec-groups/{id}` / `POST /api/spec-groups/{id}/restore`
- `PUT /api/spec-groups/{id}/spec-types` で所属、表示順、必須/推奨、既定profile、既定単位を同期する。
- `GET /api/spec-templates?spec_group_id=`
- `POST /api/spec-templates`
- `PUT /api/spec-templates/{id}`
- `POST /api/spec-templates/{id}/apply-preview` で部品登録画面向けの初期行を返す。
- `GET /api/spec-suggestions?category_ids[]=...&q=...` で部品登録向けの候補だけ返す。

## 移行方針

1. 既存 `package_groups/packages` はそのまま使い、UIと一括並び替えだけ分類内へ寄せる。
2. `spec_groups` と pivot/テンプレート系テーブルを追加する。
3. 既存 `spec_types` を初期スペック分類へ自動所属させる Seeder を作る。
4. 既存の全件 `/api/spec-types` は残しつつ、部品登録画面は `spec-suggestions` へ段階移行する。
5. `select` UI は検索ピッカーへ置換し、全件表示は管理者向け検索画面に限定する。

## 完了条件

- パッケージは分類を選ばないと一覧・並び替えできない。
- スペック項目はスペック分類内で並び替えられる。
- `電源電圧` のような共通スペックを複数スペック分類に所属させられる。
- 部品登録時、選択カテゴリに応じたスペック分類とテンプレートが提示される。
- テンプレート適用で、TR/OPアンプ/マイコンなどの代表スペック行が自動生成される。
- スペック項目選択で全件候補が一気に出ない。
