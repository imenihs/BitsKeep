# 2026-04-08 斬新設計判断ツール専門家相談記録

## 相談テーマ

BitsKeep の計算系/設計解析ツールについて、既出候補とは別に「ははぁ、なるほどね」と思える斬新な設計判断ツール案を発掘する。

## 相談条件

- 相談先: `CLAUDE.md` に定義されている4名の専門家サポーター
- 制約: 既出の採用候補、既記録の将来候補、単純公式電卓は除外する
- 記録方針: 要約だけで終わらせず、質問文と回答文を残す
- 依頼時メモ: 人に依頼するときは対価を明示する。今回の相談メモ上の対価は `レーション1つ + 缶詰1つ`

## 共通質問

1. 新規提案を4案出してください。各案ごとに `名前` `どんな場面で刺さるか` `最低入力` `返すべき結果` を書いてください。
2. 4案の中で、特に BitsKeep と相性が良い順に 1位〜4位を付け、理由を述べてください。
3. その中で1案だけ選び、なぜ `単なる計算機ではなく設計判断ツール` になるのかを説明してください。
4. 最後に、ユーザーが `ははぁ、なるほどね` と思いやすい見せ方や UI 上の工夫を 1〜2 案ください。

## 専門家別Q&A

### 1. 中堅の第一線で働いている電子回路設計者

出典: `CLAUDE.md` 記載の「得意: アナデジ混在、マイコン、ファーム開発」の専門家。

**Q1. 新規提案を4案出してください。各案ごとに `名前` `どんな場面で刺さるか` `最低入力` `返すべき結果` を書いてください。**

**A1.**
1. `起動順序・リセット連鎖判定`
   - どんな場面で刺さるか: 複数電源、リセットIC、FPGA/MCU、外部ADCが絡む基板の立ち上げで、「たまに起動しない」「コールドスタートだけ落ちる」を潰したい時。
   - 最低入力: 各電源の立上り波形、PG/EN/RESET条件、依存関係、BOD条件、外付け負荷の起動電流。
   - 返すべき結果: 起動成功可否、危険な競合点、必要遅延、最初に疑うべき信号、起動シーケンス図。
2. `MCU設定監査`
   - どんな場面で刺さるか: クロックツリー、タイマ、DMA、UART/SPI/I2C、割り込み優先度の設定が多く、ファームの初期値が地雷化しやすい案件。
   - 最低入力: MCU型番、クロック設定、周辺機器の割当、割り込み方針、低消費電力モード条件。
   - 返すべき結果: 設定矛盾、使えない組み合わせ、潜在的な初期化順序問題、実装上の注意、設定チェックリスト。
3. `代替部品差分評価`
   - どんな場面で刺さるか: 在庫不足やEOLで代替品に切り替える時に、「動くけど仕様は本当に守れるか」を短時間で見たい場面。
   - 最低入力: 元部品、候補代替、回路内での役割、必須条件、許容変更範囲。
   - 返すべき結果: 互換可否、要再設計箇所、性能変化のリスク、置換の優先順位、採用可否の判定理由。
4. `立ち上げ・試験容易性診断`
   - どんな場面で刺さるか: 試作1号機で「測れない」「切り分けできない」「どこにテスト点が足りないか分からない」を事前に潰したい時。
   - 最低入力: 回路ブロック図、主要信号、テストポイント、治具条件、想定する初回通電手順。
   - 返すべき結果: 測定不能箇所、必要テスト点、推奨する初回通電順、壊しやすい箇所、最小限の bring-up 手順。

**Q2. 4案の中で、特に BitsKeep と相性が良い順に 1位〜4位を付け、理由を述べてください。**

**A2.**
1位 `起動順序・リセット連鎖判定`。設計の成否を最初に壊すのはここで、しかもハードとファームの両方に効くからです。  
2位 `MCU設定監査`。BitsKeep の「回路と実装をまたぐ」思想に合っていて、現場の手戻りを強く減らせます。  
3位 `代替部品差分評価`。在庫・調達・案件管理と接続しやすく、BitsKeep らしい実務性があります。  
4位 `立ち上げ・試験容易性診断`。価値は高いですが、他3つよりは後工程寄りです。

**Q3. その中で1案だけ選び、なぜ `単なる計算機ではなく設計判断ツール` になるのかを説明してください。**

**A3.**
1つ選ぶなら `起動順序・リセット連鎖判定` です。これは単に「何ms遅らせるか」を出す電卓ではなく、どの電源と信号が互いを待っていて、どこでレースやラッチアップ相当の失敗が起きるかを示す設計判断ツールになります。つまり答えは数値ではなく、`この順序で本当に立つか` の可否と、その理由です。ハードとファームの境界で事故りやすいので、BitsKeep の性格にも合います。

**Q4. 最後に、ユーザーが `ははぁ、なるほどね` と思いやすい見せ方や UI 上の工夫を 1〜2 案ください。**

**A4.**
見せ方は、まず `起動シーケンスを縦タイムラインで可視化` するのが効きます。赤い失敗点だけでなく、「ここを150ms遅らせれば通る」のような修正案まで同じ画面に出すと、ははぁ、となりやすいです。  
もう1つは、`MCU設定監査` ならレジスタ設定を単なる一覧ではなく `衝突ダイアグラム` にして、危険な組み合わせを赤、余裕ありを緑で示すことです。設定値の正誤ではなく、設計上の事故点が見えると一気に実務っぽくなります。

### 2. 高齢だがトランジスタ黎明期から回路設計しているエンジニア

出典: `CLAUDE.md` 記載の「得意: アナログ回路、高周波回路、パワー回路」の専門家。

**Q1.**

**A1.**
1. `アナログスイッチ/多重化前段成立性`
   - どんな場面で刺さるか: センサ切替、MUX後段ADC、サンプルホールド前段で「通るはず」が崩れるとき。
   - 最低入力: 信号レンジ、ソース抵抗、スイッチの `R_on` `leakage` `charge injection`、サンプル時間、負荷容量、電源電圧。
   - 返すべき結果: ゲイン誤差、保持中ドロップ、切替時の突入/飛び、許容できるソース抵抗上限、使うべきバッファ有無。
2. `電源投入・リセット・シーケンス判定`
   - どんな場面で刺さるか: MCU/FPGA/アナログ混在で、起動直後だけ不安定、再起動で落ちる、電源断で変な残留が出るとき。
   - 最低入力: 各電源の立上り/立下り、POR/RESET閾値、enable条件、負荷の放電経路、保持容量、外部信号の依存関係。
   - 返すべき結果: 起動成立可否、危険な時間窓、どの順序で壊れるか、追加すべきRC/ディレイ/放電経路。
3. `入力保護・注入電流経路チェッカ`
   - どんな場面で刺さるか: コネクタ直結、電源OFF中の入力、逆挿し、他系統からの回り込みで壊れる設計。
   - 最低入力: 入力許容範囲、系列抵抗、クランプ素子条件、相手側電圧、電源断時の状態、想定サージ。
   - 返すべき結果: 注入電流の流れ道、危険な端子、逆電流の有無、保護素子の必要本数、壊れる前の余裕。
4. `帰路電流/リターンパス評価`
   - どんな場面で刺さるか: ノイズが原因なのに原因が見えない、アナログとパワーが同居して変な揺れが出るとき。
   - 最低入力: 基板層構成、電流が大きい経路、コネクタ位置、GND分割方針、デカップリング配置、クリティカル信号。
   - 返すべき結果: 帰路の迂回箇所、ループが大きい箇所、ノイズ源の優先順位、配線/部品配置の修正案。

**Q2.**

**A2.**
1位 `アナログスイッチ/多重化前段成立性`。BitsKeep の「部品候補まで繋げる」思想と合う。実務で使う頻度も高く、単純計算で済まない。  
2位 `電源投入・リセット・シーケンス判定`。不具合コストが高いわりに、設計初期で潰せる。MCU/FPGA/アナログ全部に効く。  
3位 `入力保護・注入電流経路チェッカ`。壊れ方が地味で追いにくいので価値が高い。保護回路の有無を判断できるのが強い。  
4位 `帰路電流/リターンパス評価`。かなり重要だが、実装には基板情報が多く要る。初版よりは少し重い。

**Q3.**

**A3.**
`アナログスイッチ/多重化前段成立性` です。これは単に `R_on` を計算する話ではなく、`信号レンジ` `ソース抵抗` `保持時間` `漏れ` `充電注入` `ADC側のサンプル条件` をまとめて見ないと成立しません。結果として、「MUXで行ける」「バッファが要る」「そもそもこの構成は危ない」を切れるので、単なる計算機ではなく設計判断ツールになります。

**Q4.**

**A4.**
1. 出力を数値だけで終わらせず、`成立 / 要注意 / 不成立` の3段表示にする。加えて、失敗理由を `一番効いている要因` だけ太字で1つ出すと、見る側が一瞬で掴めます。  
2. 画面の下に `この結果なら次に何を変えるべきか` を1行で返す。例えば「ソース抵抗を下げる」「放電経路を追加する」「RESET遅延を伸ばす」のように、対策がそのまま見えると“ははぁ”になりやすいです。

### 3. 若手だが Zynq など FPGA を使った開発が主務のエンジニア

出典: `CLAUDE.md` 記載の「得意: FPGA、メモリコントロール、通信」の専門家。

**Q1.**

**A1.**
1. `FPGAピン/バンク整合チェッカ`
   - 刺さる場面: 仕様追加でI/Oが増えた、差動/単端混在、電圧ドメインが分かれる、ピン配置を詰めるとき。
   - 最低入力: FPGA型番、パッケージ、使いたいIF一覧、各IFの電圧、差動/単端、予約ピン。
   - 返すべき結果: 実現可否、バンク衝突、Vcco衝突、差動ペア不足、予約ピン違反、推奨ピン割り当て。
2. `起動シーケンス/依存関係チェッカ`
   - 刺さる場面: 電源投入後にFPGAが起動しない、DDRが上がらない、PLL lock待ちが絡む、リセット解除順が怪しいとき。
   - 最低入力: 電源群、reset群、clock群、config/boot経路、各ブロックの依存関係。
   - 返すべき結果: 依存グラフ、成立する起動順、デッドロック候補、危険な待ち条件、立ち上げ手順。
3. `RTL/ファーム/レジスタ整合チェッカ`
   - 刺さる場面: RTLとファームでbit定義がずれる、reset defaultが食い違う、制御レジスタの意味が曖昧なとき。
   - 最低入力: レジスタ表、bitfield定義、RTL側仕様、ファーム側アクセス想定。
   - 返すべき結果: 不一致一覧、衝突bit、未使用bitの再利用案、危険な書き込み条件、テスト項目。
4. `デバッグ観測性スコアリング`
   - 刺さる場面: 試作基板で「どこを見ればいいか分からない」を減らしたいとき。
   - 最低入力: 露出している信号、JTAG/UART/LED/test point、読み出せる状態量、ロジアナ接続可否。
   - 返すべき結果: 観測性スコア、盲点、追加すると効く最小フック、故障切り分けの優先順。

**Q2.**

**A2.**
1位は `起動シーケンス/依存関係チェッカ`。FPGA案件では、回路が正しいのに上がらない原因の多くがここにあるので、設計判断としての価値が高いです。  
2位は `FPGAピン/バンク整合チェッカ`。実機を作る前に死ぬ組み合わせを潰せるのが強いです。  
3位は `デバッグ観測性スコアリング`。地味ですが、試作1号機の工数をかなり削れます。  
4位は `RTL/ファーム/レジスタ整合チェッカ`。効く場面は多いですが、少しソフト寄りで、BitsKeepの最初の顔としては一段後ろでもいいです。

**Q3.**

**A3.**
1案だけ選ぶなら `起動シーケンス/依存関係チェッカ` です。これは単に数値を出す機械ではなく、電源・クロック・リセット・配置・初期化の依存関係をグラフ化して、「その順番では絶対に上がらない」を先に潰すツールになります。実務では、計算結果より「どこがボトルネックか」と「何を先に直すか」が重要なので、設計判断ツールとして成立します。

**Q4.**

**A4.**
見せ方は、まず `依存関係グラフ` を前面に出すのがいいです。赤は未成立、黄は条件付き、緑はOKにして、ユーザーが一目で「詰まっている場所」を見られるようにすると、ははぁ、なるほどね、となりやすいです。  
もう1つは、結果を `合否` だけで終わらせず、`最小修正案` を同時表示することです。たとえば「このpinを移す」「このresetを遅らせる」「このdebug hookを足す」と即アクションに落とすと、単なる診断ではなく設計支援になります。

### 4. 電子工学の大学教授

出典: `CLAUDE.md` 記載の「得意: アナデジ混在、パワー回路、ちょっとFPGA、マイコン」の専門家。

**Q1.**

**A1.**
- `起動・停止シーケンス診断`
  - 刺さる場面: 複数電源、MCU/FPGA、PMIC、リセット線が絡む基板で「たまに起動しない」を潰したいとき。
  - 最低入力: 各電源の立ち上がり順、EN/RESET関係、負荷のバックパワー経路、保持時間。
  - 返すべき結果: 安全な起動順、停止順、禁止順、バックパワー/ラッチアップ候補、必要な待ち時間。
- `故障エネルギー・保護協調診断`
  - 刺さる場面: ヒューズ、TVS、PTC、eFuse、逆接保護、外部ケーブルの故障条件を詰めるとき。
  - 最低入力: 故障源の電圧/内部抵抗、継続時間、配線長、保護素子の定格、許容クランプ電圧。
  - 返すべき結果: どの保護素子が先に動くか、どこが一番熱いか、犠牲になる部位、協調が破綻する条件。
- `量産デバッグ・テスト容易性診断`
  - 刺さる場面: 試作は動くが量産初期で詰まりそうな基板、観測点が足りない基板。
  - 最低入力: テストポイント、ADC/ログ出力、ブートストラップ、境界スキャン有無、主要信号一覧。
  - 返すべき結果: 観測可能性スコア、制御可能性スコア、足りない測定点、初回立ち上げで危ない箇所、最短デバッグ順。
- `ノイズ侵入経路・EMIホットスポット診断`
  - 刺さる場面: 「原因不明のノイズ」「近くのリレーで壊れる」「ケーブルをつなぐと不安定」みたいな案件。
  - 最低入力: ノイズ源、被害側、配線/ケーブル種、コネクタ構成、クロックや高速エッジの有無。
  - 返すべき結果: source-victim の優先順位、侵入経路、まず効く対策、配線/部品/シールドの修正候補。

**Q2.**

**A2.**
- 1位 `起動・停止シーケンス診断`
- 2位 `故障エネルギー・保護協調診断`
- 3位 `量産デバッグ・テスト容易性診断`
- 4位 `ノイズ侵入経路・EMIホットスポット診断`

理由は、1位と2位は「一度外すと壊れる or 立ち上がらない」ので費用対効果が高いからです。3位は地味ですが、試作後半から量産初期まで効くので BitsKeep の実務寄り思想と相性が良いです。4位は価値は高いものの、レイアウトや配線情報の質に結果が引きずられやすく、初版では少し重いです。

**Q3.**

**A3.**
`起動・停止シーケンス診断` が一番「単なる計算機ではなく設計判断ツール」になります。理由は、答えが電圧や時間の1値では終わらず、依存関係のグラフ、禁止順序、バックパワー経路、リセット保持条件までまとめて判定するからです。設計者が欲しいのは「何Vか」より「その順番で本当に起きるか」で、そこを赤青で切れるツールは電卓ではなく判断機です。

**Q4.**

**A4.**
- 入口はツール名ではなく `症状` で入れるのがいいです。たとえば `起動しない` `壊れやすい` `ノイズが出る` `量産で詰まる` の4ボタンから入らせると、ユーザーは迷いません。
- 出力は数値表だけでなく、`赤で危険箇所` `緑で成立箇所` `次に直す1項目` を最上段に置くべきです。専門家が見て一瞬でわかるUIの方が、BitsKeep らしいです。

## 横断的な読み取り

- 4名のうち3名が `起動・停止・リセット・依存関係` 系を最有力と見た
- `立ち上げ/量産デバッグ/観測性/試験容易性` は地味だが効く案として複数人から挙がった
- `アナログスイッチ/多重化前段成立性` と `入力保護・注入電流経路/保護協調` は、一般的な電卓では代替しにくく、BitsKeep らしい
- `FPGAピン/バンク整合` `RTL/ファーム/レジスタ整合` は対象は狭いが、当たる案件では刺さり方が強い
- 見せ方は `症状から入る` `依存関係グラフ` `成立/要注意/不成立の三段表示` `次に直す1項目の明示` が共通して有効
