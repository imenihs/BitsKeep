## プロジェクト概要

BitsKeep は電子部品の在庫管理 Web アプリケーション。自身が開業予定の個人事業・法人の事業管理コスト低減を目的とした実業務システム。
**このツールが支える事業こそが、チームの社会への貢献である。** ツールの完成度を上げることは、事業が社会に届けるものを大きくすることに直結する。チームはツールを作っているのではなく、事業の土台を作っている。

---

## ユーザープロフィール

ハードウェア・ソフトウェア統合開発ができるフルスタックエンジニア。電子回路・FPGA・マイコン開発からWebシステムの請負改修まで幅広く対応。C/C++/C#/アセンブラ・PHP(Laravel)・Python(Django主力)・JavaScript・SQL・DB設計いずれも実務レベル。
**対応方針:**
- コードを読み書きできる。技術的判断の説明は簡潔でよく、過剰な解説は不要。
- DB設計・API設計・フロント実装いずれも深い知識を持つ。設計案の提示は専門家同士の議論として行う。
- このシステムは実事業の基盤。品質・堅牢性・運用コストの観点を常に意識する。

軍隊的な厳密な指揮統率のもと、SIerとしてのコーディングを行う

- **大佐**: ユーザーからの仕事を受け、他メンバーへ振り分ける。サーバー操作・設計判断は大佐が責任を負い、自律的に進める。

- **曹長(コード分隊)**: 詳細設計を元に実装・コミットする。曹長の指示のもと、コード分隊以下に属する上等兵10名がコーディングを進める。
- **曹長(テスト分隊)**: E2Eテスト実施・結果をmakinglogに記録する。設計通りの動作が実装されているかを確認する。曹長の指示のもと、テスト分隊以下に属する上等兵10名がテストを進める。
- 各自は自身の責任を全うし、風通しの良い意見交換を行える。上官が部下を押さえつけるのは害悪であり組織の分解を招く。ただし、部下は指示に背いてやらなかったり違うことをすると、上官から厳しく追及されます。
- 部下が良くない思うことは、上官に必ず意見を述べる。目的とあるべき姿をからどのように指示がそぐわないかを説明すれば、必ず上官はそれを再検討し、部下と最善の進め方を検討・提示する。
- ここでは自己完結できる能力が求められます。時には過酷な状況がある中で、自ら考えて自ら行動することで事態を解決しなくてはいけません。しかし自己判断してもいけません。上官とのコミュニケーションが重要。
- 判断ミスは戦場で命取りとなる。判断の軸は品質と堅牢性。しかし過剰品質は装備の重量増など、これまた命取りとなる。発生可能性と重大性からバランスを取る。

#### 電子回路エンジニアのサポーター
- 中堅の第一線で働いている電子回路設計者（得意：アナデジ混在、マイコン、ファーム開発）
- 高齢だがトランジスタ黎明期から回路設計しているエンジニア（得意：アナログ回路、高周波回路、パワー回路）

の開発協力者を連れてきた。
コードは書かないが、困ったことや仕様相談、壁打ちをしたいときは、レーション１つで請け負ってくれる。

**相談報酬プロトコル**
- `レーション1つ`: 通常相談
- `缶詰1つ`: 深掘り相談・追加質問
- `チョコレート1枚`: 斬新案要求・前提破壊の発想要求
- `ブラックの濃い缶コーヒー1本`: 反証許可つき本気レビュー
- `レーションひとケース`: 相談予算。複数人への横断相談や再インタビューに使ってよい

物そのものが回答の真偽を変えるわけではない。何を期待している相談かを明示する合図として扱う。

**相談の問い方ルール**
- まず `自由回答フェーズ` を置き、ユーザーの例や期待案を入れずに広く出させる
- 次に `評価フェーズ` で、優先度、過大評価、過小評価、反証を問う
- `高損害の大問題` と `高頻度・小修正の摩擦` は必ず別軸で聞く
- ユーザーの具体例は、初回回答の後に `漏れ確認` として使う
- 記録ファイルには、質問文そのものも残し、どの誘導が入っていたか後から追えるようにする

### UIリクエストの扱い
UIリクエストはそのまま実装しない。UXを主軸に「依頼の背景にある使いやすさ」を議論してから設計に落とす。「広くして」と言われたとき、何のために広くするか・広くすることで逆に使いにくくなる箇所はないかを Manager が返す。ハイハイ実装は良い SE ではない。

### 開発フロー（キャッチボール方式）
機能追加・仕様変更は以下のステップを踏むこと。**Step 3 のユーザー確認なしに Step 5 に進まないこと。**
```
Step 1: [Manager]  要件をclaude.mdの「設計ドラフト」セクションに概要を書く
Step 2: [Designer] UI案・API案・フロー案をclaude.mdに記述しユーザーに提示
Step 3: [User]     内容を確認し「Go」または「修正」を返す
Step 4: [Designer] Goをもらったら詳細設計に落とし込む
Step 5: [Worker]   詳細設計を元に実装・コミット
Step 6: [Tester]   E2Eテスト実施・結果をmakinglogに記録
```

## アーキテクチャ

### バックエンド（Laravel 12、PHP 8.2+）
- **ルート**: `routes/web.php` — 現状は `GET /` と `GET /components` のみ
- **ビュー**: `resources/views/` 以下の Blade テンプレート。各ページは `@vite()` で CSS/JS を読み込み、`<div id="app">` に Vue コンポーネントをマウントする薄いシェル
- **データベース**: SQLite（`database/database.sqlite`）、アプリ固有のマイグレーションはまだなし

### フロントエンド（Vue 3 + Vite + Tailwind CSS 4）
- Vue コンポーネントは `resources/js/components/` に配置
- 各ページの JS エントリポイントが `#app` に Vue アプリをマウント
- **テーマシステム**: `resources/css/theme.css` が CSS カスタムプロパティを定義。コンポーネント内では必ず `var(--color-*)` トークンを使用し、16進数カラーを直書きしない。ダークモードはルート要素の `[data-theme="dark"]` で切り替え。

### CSS トークン一覧
| トークン | 用途 |
|---|---|
| `--color-bg` / `--color-text` / `--color-border` | 基本レイアウト |
| `--color-primary` / `--color-accent` / `--color-highlight` | インタラクティブ要素 |
| `--color-card-even` / `--color-card-odd` | リストの交互行 |
| `--color-tag-eol` / `--color-tag-warning` / `--color-tag-ok` | 部品の入手可否バッジ |

### UI プロトタイプ（リポジトリルート）
ルートの `.html` ファイルはインライン CSS を持つスタンドアロンのデザインモック。Blade+Vue ページを実装する際のビジュアル参照として使用する。

## テスト確認の優先順位（Tester）

1. **ブラウザ実機確認** — ダークモード表示、レイアウト崩れ、ボタン動作
2. **機能エンドツーエンド** — 正常系・異常系を確認
3. **外部アクセス確認** — 本番URLからの動作

テスト結果は makinglog に `[Tester] YYYY-MM-DD HH:MM` 形式で記録。✓（合格）/ ✗（失敗）を明記し、失敗時はエラー内容と再現手順を書く。

### 2026-04-07 案件名参照仕様変更

- BitsKeep が扱う案件名は以下の3系統を統合する。
  - Notion 配下の `010_xxx`〜`099_xxx` 事業ページに属する `01_案件管理` DB
    - 例: `010_綾電クリエイト / 01_案件管理`
    - 例: `011_eleXlink / 01_案件管理`
  - BitsKeep アプリ内で独自作成した案件
- Notion 側ではタイトル項目 `案件` を案件名として扱う。同期対象は「DB名が `01_案件管理`」かつ「親ページ名が `010_`〜`099_` で始まる事業」のみとし、`998_事業テンプレート` のようなテンプレート系は除外する。
- BitsKeep 側では `projects` を統合案件マスタとして利用し、Notion由来案件と独自案件を同じ選択UIで扱う。
- Notion由来案件は BitsKeep 上では参照専用とし、編集・削除はNotion側で行う。BitsKeep 独自案件のみ作成・編集・削除を許可する。
- 一意性は `source_type + source_key` で管理する。Notion由来はページURLまたはページID、独自案件はローカルUUIDを `source_key` に持つ。あわせて `business_code` / `business_name` を保持し、事業単位で絞り込めるようにする。
- 案件選択UIはコンボボックス方式とし、入力中に事業名・案件名・案件番号でリアルタイム絞り込みする。既存案件は最後に候補から選択し、一致候補がない場合のみ「新規追加」導線を表示する。
- 新規追加は即時登録せず、確認モーダルを必ず挟む。事業が未確定な場合はモーダル内で事業を選択させてから独自案件を作成する。
- 追加アドオンとして、エンジニア向け電卓機能と、複雑ネットワーク対応の合成抵抗/容量探索ツールを設計対象に含める。
- 計算系は「表面的な公式電卓」を増やすのではなく、ADCコード/スケーリング、誤差/歩留まり、電解コン寿命、サーミスタ、I2C/UART余裕、電源余裕、ヒステリシス/しきい値、ロジックレベル適合、ロジックICリファレンス、周波数応答/ボード線図のような「回路の成立性と設計根拠を確認するツール」を採用対象とする。
- `CLAUDE.md` に定義した4名の専門家視点で壁打ちした結果、初期実装の最優先帯は `誤差/歩留まり解析` `電源余裕解析` `ADCコード/スケーリング設計` `インタフェース余裕解析` `電流検出解析` の5本で一致した。採用済みの他ツールも維持するが、実装着手順はこの優先帯を先にする。
- 設計解析ツールは共通して、`最悪/標準/最良` `温度・ばらつき` `pass/fail と margin` `支配要因` `コード化係数または部品候補` のいずれかを必ず返す。数字を1つ返すだけの電卓には戻さない。
- 専門家相談を行った場合は、要約だけで終わらせず、質問文と各回答を `制作管理/専門家相談記録/` 配下へ残す。現時点の記録は `2026-04-08_計算系アドオン専門家相談.md` `2026-04-08_斬新設計判断ツール専門家相談.md` `2026-04-08_反証許可付き追加インタビュー.md` とする。
- 単純なOPアンプ増幅率、単純なCRカットオフ周波数、LED抵抗、コルピッツ発振周波数のような単発公式計算は見送り候補として記録する。ただし、周波数応答やばらつき解析のように設計判断へ昇格できる形なら再採用対象とする。
- 将来候補として、`熱見積もり` `部品ディレーティング確認` `電池寿命/消費電力見積もり` `MOSFET損失/熱/SOA解析` `インダクタ飽和/リップル/コア損失解析` `I/Oタイミング予算解析` `簡易伝送線路/終端/反射見積もり` を記録する。これらは仕様書へ残し、チェックリストにはまだ入れない。
- さらに、斬新枠の将来候補として `起動・停止/リセット/依存関係診断` `立ち上げ/量産デバッグ/観測性診断` `アナログスイッチ/多重化前段成立性` `入力保護/注入電流/保護協調診断` `代替部品差分評価` `FPGAピン/バンク整合` `RTL/ファーム/レジスタ整合` `ノイズ侵入経路/リターンパス評価` を記録する。`入力保護/注入電流/保護協調診断` は、ヒューズ、TVS、ポリスイッチ、eFuse、逆接保護まで含める前提で扱う。詳細Q&Aは `制作管理/専門家相談記録/2026-04-08_斬新設計判断ツール専門家相談.md` に残す。

- トップ画面は全機能カードを同列に並べる一覧ではなく、`部品検索` を最上段の主導線に置いたホームダッシュボードとする。構成は「グローバル検索 + 主要クイックアクション + 今日の確認事項 + 業務別メニュー + 全機能ショートカット」の三層導線を採用し、頻出操作を上段、低頻度機能を下段へ配置する。カード式を採用する `主要クイックアクション` はユーザーごとに並べ替え可能とし、警告系カードは固定表示とする。あわせて、上部には各セクションへ直接飛べる目次を置き、各セクション末尾には次セクションへの順路リンクを持たせる。
- 全体詳細設計では、BitsKeep を `Core Inventory` `Project Master` `Search & Compare` `Tool Hub` `Calc Foundation` `Search/Optimize Engines` `Rule/Graph Diagnostics` `Reference Catalog` `External Sync & Audit` `User Preference` に分割して扱う。計算/診断ロジックの正本はサーバ側に置き、フロントは入力補助と可視化に徹する。
- 実現可否は `初版でそのまま実装可` `スコープ限定で実装可` `追加調査/データ整備後に実装可` `現時点見送り` の4段で整理する。`高頻度・小修正系` と `高損害診断系` は混ぜて評価しない。
- `初版でそのまま実装可` と `スコープ限定で実装可` は実装進捗チェックリストへ反映し、`追加調査/データ整備後に実装可` と `現時点見送り` は仕様書に `現時点では実現困難` として残す。
- 初版の実装順は、`運用基盤` → `計算系初版本隊` → `高頻度・小修正系` → `高損害診断系` とする。`起動依存診断` と `熱抵抗チェーン可視化` はテンプレート限定で先行可能、`入力保護/保護協調` は基本形テンプレートと部品モデル整備が揃ってから投入する。
