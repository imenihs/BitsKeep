# BitsKeep 部品在庫管理システム 仕様書

---

## 1. 目的（絶対にブレてはいけない軸）

> **電子回路設計者が「使うべき部品」を最短で判断できるようにすること**

これは単なる在庫管理ではない。

| NG 思想 | 正しい思想 |
|---|---|
| 在庫があるものを探す | 設計要件に適合する部品を見つける |
| 安いものを探す | 価格・入手性・実装性で最適を判断する |

---

## 2. ユーザー種別と権限

| 種別 | 権限 |
|---|---|
| 閲覧者 | 検索・閲覧・データ出力のみ |
| 編集者 | 入出庫・部品編集 |
| 管理者 | マスタ編集・ユーザー管理 |

- ログインIDは当面 `メールアドレス` を維持する。
- 表示名は `users.name` を使い、画面上の呼称やユーザーメニューにはメールアドレスではなく表示名を優先表示する。
- `メールアドレス` はログインIDに加え、パスワード再設定、招待、将来の通知・確認メールの送信先として扱う。
- SNSログインは別レイヤの認証手段として将来追加を検討する。初期候補は `Google` と `GitHub` とし、ローカル認証を廃止せず併存させる。
- SNSログインを導入する場合は、`verified email` を軸に既存ユーザーへ安全に紐付ける。認証プロバイダ情報は `user_auth_providers` のような別テーブルで管理する。

---

## 3. 設計者の思考フロー（UI設計の基準）

UIはこの順序で情報を提示すること。**在庫は最後。**

1. スペック（電気的条件を満たすか）
2. 実装可否（パッケージ・サイズ）
3. データシート（設計確証）
4. 価格（コスト感覚での取捨選択）
5. 在庫（試作・手配の即時可否確認）

- 棚卸機能
- 使用履歴（日時・出庫者）
- 在庫警告機能（部品ごとの在庫下限設定）
- ユーザー管理

## 6. データモデル

### 設計方針

| 管理内容 | 採用パターン |
|---|---|
| 多対多の関係 | 中間テーブル |
| 数値スペック（定格・寸法等） | `component_specs`（spec_typeで区別） |
| 不定型・自由入力項目 | `component_attributes`（キーバリュー） |
| 在庫の物理単位 | `inventory_blocks`（ロット・リール単位） |
| 入出庫履歴 | `transactions` |
| 外部案件ソース統合 | `projects` に `business_code` / `business_name` / `source_type` / `source_key` を持たせて統合管理 |
| 操作監査 | `audit_logs` |

`components` は最小情報（型番・名称程度）で登録可能。段階的入力を前提とする。

### テーブル一覧

| テーブル名 | 用途 |
|---|---|
| `components` | 部品基本情報（型番・通称・在庫数・入手可否・ファイルパス等） |
| `component_specs` | スペック値（定格・寸法・電圧・電流等、単位付き） |
| `spec_types` | スペック種別と単位の定義マスタ（例: 電圧 + V/mV/kV） |
| `component_attributes` | 柔軟な属性追加項目（キーバリュー） |
| `attribute_keys` | キーバリュー属性で使うkey候補の定義 |
| `categories` / `component_category` | 分類（多対多） |
| `package_groups` / `packages` / `components.package_id` | パッケージ分類と詳細パッケージ。1部品は1詳細パッケージを持つ |
| `projects` / `component_project` | 統合案件マスタ（`010_xxx`〜`099_xxx` 事業配下の Notion案件 + 独自案件）と部品紐付 |
| `locations` / `component_location` | 保管棚・QRコード対応の場所管理（階層構造あり） |
| `suppliers` | 商社マスタ |
| `component_suppliers` | 部品ごとの仕入情報（単価・商社型番・URL） |
| `supplier_price_breaks` | 数量条件ごとの単価定義（価格ブレーク） |
| `shipping_rules` | 商社ごとの送料条件（閾値・送料） |
| `inventory_blocks` | 在庫単位（パッケージ種別・ロット番号・リールID等） |
| `stock_orders` | 発注記録（部品・商社・数量・状態・発注日等） |
| `transactions` | 入出庫履歴（誰が・いつ・どれだけ・理由） |
| `component_altium_links` | Altium連携情報（sch/pcbライブラリパス・名称） |
| `users` | ユーザー情報（表示名・役割・認証情報） |
| `audit_logs` | 操作履歴ログ（変更差分をJSON形式で記録） |

### ファイル保存の成功条件

- 画像・データシートの保存成功は、`DBの path 更新` ではなく `実ファイルが保存先に存在すること` を含めて判定する
- 保存先ディレクトリへ書き込めなかった場合は、DB更新をロールバックし、UIへエラー応答を返す
- つまり「ファイルが存在しないのに path だけ登録される状態」は許容しない

### 入力モーダルの閉じ方

- 情報入力を伴うモーダルは、画面外クリックだけで閉じてはならない
- 閉じる操作は `×` `キャンセル` `明示的な閉じるボタン` に限定する
- 未保存変更がある入力モーダルは、閉じる前に確認を出す

### カスタムフィールドの保存条件

- カスタムフィールドは `key/value` の両方が埋まっている行のみ保存対象とする
- `key` の重複は許可しない
- 空行や片側だけ埋まった行で保存しようとした場合は、UI側で保存不可にする

### 主要テーブル定義

```sql
-- 部品基本情報

  id, manufacturer, part_number, common_name,
  condition,             -- 新品/中古
  quantity_new, quantity_used,
  procurement_status,    -- EOL / Active 等
  datasheet_path, image_path,

-- 在庫単位（ロット）
inventory_blocks (

  deleted_at, created_at, updated_at
)

-- 発注記録
stock_orders (
  id, component_id (FK), supplier_id (FK),
  quantity, status,  -- pending / received / cancelled
  order_date, expected_date, received_date,
  created_by (FK→users), created_at, updated_at
)

-- 入出庫履歴
transactions (
  id, component_id (FK), inventory_block_id (FK),
  user_id (FK), type,  -- in/out
  quantity, reason, created_at
)

-- Altium連携
component_altium_links (
  id, component_id (FK),
  sch_lib_path, sch_lib_name,
  pcb_lib_path, pcb_lib_name,
  created_at, updated_at
)

-- 統合案件マスタ
projects (
  id, name, description,
  business_code,         -- 010..099 or local-only code
  business_name,         -- 綾電クリエイト / eleXlink 等
  source_type,           -- notion / local
  source_database_key,   -- Notion database or data source ID
  source_key,            -- Notion page ID or local UUID
  external_code,         -- 案件番号など外部識別子
  external_url,          -- Notion page URL（localはNULL）

  deleted_at, created_at, updated_at

  id, component_id (FK), project_id (FK),
  linked_transaction_id (FK),  -- 出庫起点の紐付けなら設定

  created_at, updated_at
)

-- 操作ログ
audit_logs (
  id, model_type, model_id, user_id,
  action, field_changes,  -- JSON形式
  created_at
)
```

### 発注追跡（stock_orders）の動作仕様

発注リストの送信後も、どの部品・商社の発注がいつ完了したか・キャンセルされたかを追跡できるようにするための機能。

#### データフロー
1. **在庫警告画面で部品選択 → 発注リスト作成**
   - 部品チェックボックスで選択 → 「発注リストに追加」で草稿作成
   - ローカルストレージに `orderDraft` として保持
   - 発注済み部品（stock_orders.status='pending'）は選択不可にする

2. **発注リスト送信**
   - 商社別グループを確認 → 「発注」ボタンで POST /stock-orders
   - 各部品1行ごとに stock_orders レコード作成（`status='pending'`）
   - 送信後、orderDraft をクリア（localStorage 削除）

3. **発注状態の確認**
   - 在庫警告画面で「発注済み」バッジを表示（stock_orders で pending/received の判定）
   - GET /stock-orders/component/{id}/pending で該部品の pending 発注を取得
   - 同じ商社への重複発注を防止

4. **発注受取・キャンセル**
   - 発注商社から納品が来たら、管理者が stock_orders の `status` を `received` に変更
   - 発注をキャンセルする場合は `status` を `cancelled` に変更
   - 状態変更時は `received_date` または `expected_date` を更新

#### UI上の見え方
- **在庫警告一覧**: 既に pending 発注がある部品の行に「⏳ 発注待ち」バッジを付与
  - バッジにマウスオーバーで「商社X: 数量10 (2026-04-20 注文)」などを表示
- **部品詳細**: 「発注状態」セクションで pending/received/cancelled の一覧を表示
- **発注モーダル**: 新規追加前に「この部品は既に商社Xへ20個発注待ちです」と警告

#### API エンドポイント
| エンドポイント | メソッド | 説明 |
|---|---|---|
| `/stock-orders` | GET | 発注一覧（フィルタ対応） |
| `/stock-orders` | POST | 発注作成 |
| `/stock-orders/{id}` | PATCH | 発注状態更新（status/dates） |
| `/stock-orders/{id}` | DELETE | 発注削除（キャンセル扱い） |
| `/stock-orders/component/{component_id}/pending` | GET | 該当部品の pending 発注取得 |

## 9. UI設計ルール

### CSS トークン一覧
| トークン | 用途 |
|---|---|
| `--color-bg` / `--color-text` / `--color-border` | 基本レイアウト |
| `--color-primary` / `--color-accent` / `--color-highlight` | インタラクティブ要素 |
| `--color-card-even` / `--color-card-odd` | リストの交互行 |
| `--color-tag-eol` / `--color-tag-warning` / `--color-tag-ok` | 部品の入手可否バッジ |

### UI プロトタイプ（リポジトリルート）
`詳細仕様/UI構成/xxx .html` ファイルはインライン CSS を持つスタンドアロンのデザインモック。
Blade+Vue ページを実装する際のビジュアル参照として使用する。

### 全体共通
- ライト・ダークテーマ対応。カラーは `theme.css` の `var(--color-*)` トークンのみ使用。
- `null` 項目は表示しない。
- カード形式表示。背景は `card-even` / `card-odd` で交互塗り分け。
- モーダルウインドウと、そのESC挙動はテンプレート化されているので、それを利用してモーダルウインドウを作ること

### トップ画面（ホームダッシュボード）
- トップ画面は「全機能を同列カードで並べる索引」にしない。最優先導線は `部品検索` とする。
- 最上段には部品・案件・棚・機能を横断するグローバル検索を置き、キーボード入力から直接目的へ到達できるようにする。
- 主要クイックアクションは 4〜6 件に制限し、利用頻度が高いものだけを上段に置く。低頻度機能は下段の業務別メニューへ送る。
- 主要クイックアクションのカード順はユーザーごとに並べ替え可能とする。保存単位はユーザー単位を基本とし、必要に応じて役割またはモード別の保持も許容する。
- 並べ替えは通常クリックと競合させない。明示的な「並び替えモード」に入った時だけドラッグ&ドロップ可能にする。

## 13. DB設計修正（確定）

### spec_types / spec_units / 接頭辞ポリシーの分離

`spec_types` はスペックの意味と基底単位を持つ。`spec_units` は明示的な単位候補や倍率を持つ。加えて、スペック種別ごとに「候補へ出す接頭辞」「表示時に優先する接頭辞」「表記ゆれ吸収ルール」を設定できるようにする。

単位入力はドロップダウンで縛らず、テキスト入力を基本とする。ドロップダウンや datalist は入力補助であり、保存可否の唯一の基準にしてはならない。

```sql
-- 修正前（誤）
spec_types ( id, spec_name, unit, description )

-- 修正後（正）
spec_types ( id, spec_name, base_unit, description, prefix_policy )
spec_units ( id, spec_type_id (FK), unit_label, factor )
-- 例: spec_type_id=1(voltage) → V, mV
--     spec_type_id=2(resistance) → Ω, kΩ, MΩ
```

#### 接頭辞ポリシー

`prefix_policy` は JSON などの構造化値として持つ。初版は列追加または補助テーブルのどちらでもよいが、UI/API では次の概念を扱えることを要件とする。

| 項目 | 役割 | 例 |
|---|---|---|
| `base_unit` | 正規化先の基底単位 | `Ω`, `A`, `V`, `F`, `Hz` |
| `input_prefixes` | 値欄・単位欄で許容する接頭辞 | `M,k,,m,u,n,p` |
| `suggest_prefixes` | datalist に出す接頭辞 | 抵抗値なら `M,k,,m` |
| `display_prefixes` | 表示時の自動逆変換で使う接頭辞 | 電流なら `A,m,u,n` |
| `unit_aliases` | 表記ゆれ吸収 | `Ω -> Ω`, `ohm -> Ω` |
| `prefix_aliases` | 接頭辞ゆれ吸収 | `K -> k`, `µ/μ -> u` |
| `allow_free_unit` | 候補外単位の手入力を許すか | 原則 `true` |

接頭辞ポリシーは入力制限ではなく、主に **候補表示・正規化・表示逆変換の品質を上げるための設定** とする。候補にない `uΩ` が入力されても、`base_unit=Ω` かつ接頭辞として解釈可能なら保存できる。候補外を完全拒否すると、シャント抵抗、リーク電流、RF、小信号などの例外を壊すためである。

代表設定例:

| spec_type | base_unit | suggest/display prefixes | 備考 |
|---|---|---|---|
| 抵抗値 | `Ω` | `M,k,,m` | 候補は実務頻出へ絞る。`uΩ` 等は自由入力で許容 |
| 容量 | `F` | `u,n,p` | `mF` は候補に出してもよいが優先度低 |
| 電圧 | `V` | `k,,m,u` | `K` は接頭辞としては `k` に補正 |
| 電流 | `A` | `,m,u,n` | リーク電流向けに `pA` も設定可 |
| 周波数 | `Hz` | `G,M,k,` | `KHz` は `kHz` に補正 |
| 温度 | `°C` | なし | 接頭辞を使わない |
| 許容差 | `%` | なし | 接頭辞を使わない |
| チャンネル数 | `ch` | なし | 接頭辞を使わない |

### component_specs の数値表現を「人間向け表示 + value_profile + 基底単位正規化」へ再設計

#### なぜ変えるか

従来の `component_specs.value_numeric` 単独方式は、`数値検索のための裏値` と `人が見る値` を無理やり 2 列へ押し込んでいたため、次の問題を抱えていた。

1. `1uA` と入力したいのに、検索都合で `0.000001` を別欄へ手入力させる必要がある  
2. `2〜5V` のような範囲値を自然に扱えない  
3. 保存された `value` が人間向け表示、`value_numeric` が検索用内部値という二重構造になり、入力・表示・検索の意味がズレやすい  
4. `1e-6A` で検索しても、表示が `1uA` のデータと整合しないケースが起きやすい  

電子部品のデータシートと人間の作業導線は `uA / mA / kΩ / ns` のような接頭語付き表記を前提にしている。したがって BitsKeep も、**入力は人間に合わせ、内部比較だけを基底単位へ統一する** 方針へ切り替える。

ただし、`single / range + 中央値` だけで全部を表す設計も不十分である。電子部品のデータシートでは、`typ` は代表値、`min/max` は保証境界や条件端であり、意味が違う。たとえば `typ=5mA, max=10mA` を単純な `5〜10mA` の範囲に潰すと、設計時に不要なマージンを要求したり、逆に typ を保証値のように誤読したりする。

そのため、BitsKeep では **`null の入り方で意味を推定しない`**。スペックの意味は `value_profile` で明示し、数値列はその profile に従って埋める。

#### 採用方針

- 入力:
  - 人は `1uA` 相当の値を、`value=1` `unit=uA` のように読みやすい形で入力する
  - 値の意味は `value_profile` で明示選択する
  - profile は `typ / range / max_only / min_only / triple` の5種とする
- 内部保存:
  - すべて `spec_types.base_unit` に正規化して保存する
  - 例: `1uA -> 0.000001 A`、`10kΩ -> 10000 Ω`
- 表示:
  - 保存後は内部値から最も読みやすい接頭語へ戻して表示する
  - 例: `0.000000001 A -> 1nA`
- 検索:
  - 検索時も同じ正規化ルールを通し、入力単位に依存しない比較を行う
  - ただし `typ / range / max / min` は意味が違うため、検索も profile を意識して照合する

#### value_profile の定義

| `value_profile` | 意味 | 埋める列 |
|---|---|---|
| `typ` | 1点の代表値 | `value_numeric_typ` |
| `range` | 最小〜最大の範囲 | `value_numeric_min`, `value_numeric_max` |
| `max_only` | 上限のみ | `value_numeric_max` |
| `min_only` | 下限のみ | `value_numeric_min` |
| `triple` | min / typ / max の3値 | `value_numeric_min`, `value_numeric_typ`, `value_numeric_max` |

`min/max/typ` の `NULL` パターンだけで profile を推定してはならない。  
判定の正本は常に `value_profile` とする。数値列の `NULL` は「その意味の値を持たない」ことだけを表す。

#### DB 変更方針

既存データを消さず、`component_specs` に列追加で移行する。現行の `value_mode` / `value_numeric` は第1段階実装の互換列として残っているが、最終的な正本は以下へ移す。

```sql
component_specs (
  id,
  component_id (FK),
  spec_type_id (FK),
  value,              -- 人が見る値。例: "1" / "2 〜 5"
  unit,               -- 人が見る単位。例: "uA" / "V" / "kΩ"
  value_profile,      -- "typ" / "range" / "max_only" / "min_only" / "triple"
  value_numeric_typ,  -- typ値（基底単位）
  value_numeric_min,  -- 最小値（基底単位）
  value_numeric_max,  -- 最大値（基底単位）
  normalized_unit     -- 基底単位。例: "A" / "V" / "Ω"
)
```

#### 各列の役割

| 列 | 役割 | 例 |
|---|---|---|
| `value` | 人が読む表示値 | `1` / `2 〜 5` / `1 / 5 / 10` |
| `unit` | 人が読む表示単位 | `uA` / `kΩ` / `ns` |
| `value_profile` | 値の意味を明示する列 | `typ` / `max_only` / `triple` |
| `value_numeric_typ` | 代表値（基底単位） | `0.000001` / `5` |
| `value_numeric_min` | 下限値（基底単位） | `2` / `0.0008` |
| `value_numeric_max` | 上限値（基底単位） | `5` / `10` |
| `normalized_unit` | 内部比較に使う基底単位 | `A` / `V` / `Ω` |

#### 正規化ルール

登録・更新時はサーバー側で以下を自動計算する。クライアントから送られた内部値は信用せず、毎回再計算する。

| spec_type | 基準単位 | 入力例 | 内部値 |
|---|---|---|---|
| resistance | Ω | `10kΩ` | `10000 Ω` |
| voltage | V | `100mV` | `0.1 V` |
| current | A | `50mA` | `0.05 A` |
| power | W | `500mW` | `0.5 W` |
| frequency | Hz | `16MHz` | `16000000 Hz` |
| time | s | `5ns` | `0.000000005 s` |

値欄・単位欄のどちらに接頭辞が入っても同じ内部値へ正規化する。

| 入力値 | 入力単位 | spec_type.base_unit | 内部値 |
|---|---|---|---|
| `1` | `kΩ` | `Ω` | `1000 Ω` |
| `1k` | `Ω` | `Ω` | `1000 Ω` |
| `1kΩ` | 空欄 | `Ω` | `1000 Ω` |
| `1KΩ` | 空欄 | `Ω` | `1000 Ω` |
| `0.001` | `MΩ` | `Ω` | `1000 Ω` |
| `1u` | `A` | `A` | `0.000001 A` |
| `1µA` | 空欄 | `A` | `0.000001 A` |

正規化パーサは `spec_types.prefix_policy` を参照し、文脈に応じて表記ゆれを吸収する。たとえば抵抗値・電圧・電流・周波数では `K` を `k` として扱ってよいが、温度やバイト系の値では無条件に同じ補正をしてはならない。

表示時は `value_numeric_* + normalized_unit` を、同じポリシーの `display_prefixes` で人間向け表記へ戻す。`1000000 Ω` のような表示は避け、抵抗値なら `1MΩ`、電流なら `1uA` のように表示する。保存済みの `unit` は「入力時の人間向け単位」として保持してよいが、一覧・詳細・比較・検索結果の表示正本は内部値からの逆変換結果を優先する。

#### profile ごとの保存ルール

**1点値（typ）**

```sql
value              = "1"
unit               = "uA"
value_profile      = "typ"
value_numeric_typ  = 0.000001
value_numeric_min  = NULL
value_numeric_max  = NULL
normalized_unit    = "A"
```

**範囲値（range）**

```sql
value              = "2 〜 5"
unit               = "V"
value_profile      = "range"
value_numeric_typ  = NULL
value_numeric_min  = 2
value_numeric_max  = 5
normalized_unit    = "V"
```

**最大のみ（max_only）**

```sql
value              = "10"
unit               = "mA"
value_profile      = "max_only"
value_numeric_typ  = NULL
value_numeric_min  = NULL
value_numeric_max  = 0.01
normalized_unit    = "A"
```

**最小のみ（min_only）**

```sql
value              = "1.8"
unit               = "V"
value_profile      = "min_only"
value_numeric_typ  = NULL
value_numeric_min  = 1.8
value_numeric_max  = NULL
normalized_unit    = "V"
```

**3値（triple）**

```sql
value              = "1 / 5 / 10"
unit               = "mA"
value_profile      = "triple"
value_numeric_typ  = 0.005
value_numeric_min  = 0.001
value_numeric_max  = 0.01
normalized_unit    = "A"
```

#### スペック種別の粒度とprofile表示

`spec_types` は「検索・比較したい仕様項目」を正本として持つ。粒度の基準は下記の通り。

- **粗すぎる（不可）**: `電流 [A]` — 単位と意味が重なり、`コレクタ電流` `ベース電流` を区別できない
- **細かすぎる（不可）**: `最大コレクタ電流` — `value_profile=max_only` と重複し、マスタが際限なく増殖する
- **適切**: `コレクタ電流`、`パルスコレクタ電流`、`コレクタ-ベース間電圧`、`直流電流増幅率` — 仕様項目単位

`value_profile` の違い（max/min/typ）は **`spec_type` を分裂させず、`value_profile` で表現する**。  
`コレクタ電流` と `パルスコレクタ電流` のように検索・比較で別軸にしたい項目は、別 `spec_type` として持つ。

`spec_types` は日本語名・英語名・記号を持ち、表記ゆれは alias で吸収する。

```sql
spec_types (
  id,
  name,            -- 互換用。原則 name_ja と同じ値を入れる
  name_ja,         -- 正本日本語名
  name_en,         -- 英語名
  symbol,          -- h_FE, V_CBO など。_ は下付き、~ は上付き表示
  suggest_prefixes, -- 入力候補接頭辞 JSON配列。null=汎用SI候補
  display_prefixes, -- 表示逆変換接頭辞 JSON配列。null=自動選択
  base_unit,
  description,
  sort_order
)

spec_type_aliases (
  id,
  spec_type_id,
  alias,      -- VCBO, Collector Base Breakdown Voltage 等
  locale,
  kind,
  sort_order
)
```

記号はDBへHTMLを保存せず、プレーンテキスト記法で持つ。`_` の後は下付き、`~` の後は上付きとして表示レンダリングする。例: `h_FE`、`V_CBO`、`f_T`、`10~6`。

#### profile のバッジ表示

`value_profile` は種別名に **前置き合成しない**。`最大コレクタ電流` のような合成文字列は生成しない。

理由: 画面に出る文字列がマスタに存在しないと、3年後の自分がマスタを探して見つからず混乱する。UIの構造がそのまま仕様の説明になっている設計を優先する。

| spec_type.name | profile | 表示 |
|---|---|---|
| `コレクタ電流` | `typ` | `コレクタ電流` |
| `コレクタ電流` | `max_only` | `コレクタ電流` + `[最大]` バッジ |
| `コレクタ電流` | `min_only` | `コレクタ電流` + `[最小]` バッジ |
| `コレクタ電流` | `range` | `コレクタ電流` + `[範囲]` バッジ |
| `コレクタ電流` | `triple` | `コレクタ電流` + `[min/typ/max]` バッジ |

バッジは種別名の右横に小さく添える。詳細・一覧・編集モーダルで共通。

#### UI 方針

- 登録画面 / 詳細編集:
  - スペック行の自由な `名前` 入力欄は置かない。保存名の正本は `spec_type_id` とする
  - `種別` はドロップダウンで選択し、候補に無い場合は `+` からスペック種別追加モーダルを開く
  - モーダルで追加した種別は即座に候補へ反映し、編集中の行で選択済みにする
  - 値の形はセグメントボタンで `typ / 範囲 / 最大 / 最小 / 3値` を明示選択する
  - 選択した profile に応じて入力欄を切り替える
  - `数値化` 手入力欄は廃止し、`基底単位換算` を自動表示へ変える
  - 値欄・単位欄の入力中に、解釈結果を小さなプレビューとして即時表示する
  - プレビューには少なくとも `基底値` と `表示値` を出す。例: `基底: 1000 Ω / 表示: 1kΩ`
  - パース不能、候補外単位、接頭辞の曖昧さがある場合は、保存前に同じ場所へ控えめな警告を出す
- 解析候補レビュー:
  - AIの抽出名は補助表示に留め、保存正本にはしない
  - `name_ja / name_en / symbol / alias` と既存 `spec_types` を照合し、同じ意味の日本語・英語・略記号を同じ `spec_type_id` へ寄せる
  - `値 / 単位 / スペック種別` に加えて `value_profile` と `基底値` プレビューを表示する
  - 適用前に `typ / range / max / min / triple` を確認・修正できるようにする

#### AI抽出での修飾語処理方針

データシートはメーカー横断で表記ゆれがある。同一の概念が `コレクタ最大電流`、`最大コレクタ電流`、`Collector Current (max)`、`I_C(max)` のように記述される。

**プロンプト側での強制分離（主防線）：**  
抽出プロンプトで「最大/最小/Max/Min/定格/連続/パルス等の境界条件修飾語を種別名から除去し、`value_profile` と `condition` に振り分けて返せ」と明示指示する。

```
入力: "コレクタ最大電流 200mA"
出力: { name: "コレクタ電流", value_profile: "max_only", value: "200", unit: "mA" }
```

**マッチング層での自動救済（副防線）：**  
`SpecTypeMatchingService` の照合失敗時、修飾語（最大・最小・Max・Min・定格等）を除去した正規化名で再照合し、ヒットした場合は自動的に該当種別へ寄せ `value_profile` を補正する。人間の判断を挟まず自動適用する。二重登録（`コレクタ最大電流` と `コレクタ電流` が別種別として並存）をシステムが防ぐ。
- 表示画面:
  - 詳細・比較・一覧で出す値は人間向け表記を優先する
  - 内部の基底値は検索・比較・補助情報に使う

#### UI 入力切替の詳細

| profile | UI 入力欄 |
|---|---|
| `typ` | `typ` 1欄 |
| `range` | `min` + `max` |
| `max_only` | `max` 1欄 |
| `min_only` | `min` 1欄 |
| `triple` | `min` + `typ` + `max` |

選択中の profile に応じて、確認欄のバッジ表示が `[最大]` `[最小]` `[範囲]` へ即時切り替わる。前置き合成（`最大コレクタ電流` のような文字列生成）は行わない。

#### リアルタイム正規化プレビュー

スペック入力欄は自由テキストを許容するため、入力中に「システムがどう解釈したか」を必ず近接表示する。これは説明文ではなく、入力結果の状態表示として扱う。

表示例:

| 入力値 | 単位 | プレビュー |
|---|---|---|
| `1k` | `Ω` | `基底: 1000 Ω / 表示: 1kΩ` |
| `1KΩ` | 空欄 | `基底: 1000 Ω / 表示: 1kΩ` |
| `4.7u` | `F` | `基底: 0.0000047 F / 表示: 4.7uF` |
| `1000000` | `Ω` | `基底: 1000000 Ω / 表示: 1MΩ` |

プレビューは登録画面、部品詳細のスペック編集、解析候補レビュー、パラメータ検索の詳細条件で共通のコンポーネントにする。バックエンド保存時の `SpecValueNormalizerService` と、フロントプレビューのパーサは同じ仕様に従い、保存後にプレビューと実保存値が食い違わないようにする。

#### 既存データの移行

既存行は削除しない。移行は必ず加算 migration で行い、現在の `component_specs` 内容を失わないことを最優先とする。

移行ルールは以下とする。

- 旧 `single` 行、および旧 `value_numeric` しか持たない行:
  - `value_profile = 'typ'`
  - `value_numeric_typ = 旧 value_numeric`
  - `value_numeric_min = NULL`
  - `value_numeric_max = NULL`
- 旧 `range` 行:
  - `value_profile = 'range'`
  - `value_numeric_typ = NULL`
  - `value_numeric_min = 旧 value_numeric_min`
  - `value_numeric_max = 旧 value_numeric_max`
- `normalized_unit` は `spec_types.base_unit` があればそれを使い、無ければ既存 `unit` を保持する
- 旧 `value_mode` / `value_numeric` は移行完了まで互換列として残してよいが、新規ロジックの判定正本には使わない

これにより、**今あるデータを消さずに** 新旧ロジックを共存させたまま段階的に移行できる。

**参照プロト：** `詳細仕様/UI構成/index.html`

```
[ヘッダ] ブランド / ワークスペース状態 / テーマ切替
```

  - `部品を探す`
  - `新規部品を登録`
  - `在庫警告を見る`

  - `抵抗/容量ネットワーク探索`

- セクション末尾に `全機能ショートカットへ` リンクを置く

**ユーザビリティ上の判断：**
- 15個前後の機能を同じ重みで並べると、選択コストが高く初手で止まる

- 警告や同期状態を同時に見せることで、ただのメニュー画面ではなく運用判断画面として機能させる
- 画面上部の目次で一気に飛べる導線と、各セクション末尾の順路リンクで迷わず読み進める導線を併用する

#### SCR-003 拡張: PDFデータシート解析ヘルパー（ChatGPT Web 自動解析を主導線とする）

**目的：** データシートからの情報抽出は `ChatGPT 画面での手貼り` が最も面倒なので、普段の運用は `PDF選択 -> 解析開始 -> 結果レビュー -> 保存` の4ステップへ圧縮する。  
ただし電子部品データシートは誤抽出コストが高いので、**結果は必ずレビューしてから適用** し、自動入力だけで即保存はさせない。

**運用モード（優先順）：**
1. `Tampermonkey + ChatGPT Web 自動解析`
2. `ChatGPTから貼り付け` モーダル（自動化失敗時の fallback）
3. `Gemini 直結解析`（API 利用を許容する環境だけの任意モード）

**プロンプトの単一ソース：**
- ChatGPT へ渡す指示文は `/web/documents/BitsKeep/プロンプト/データシート解析プロンプト.md` を唯一の正本とする
- フロントや userscript に別の prompt 文字列をベタ書きしない
- サーバー側で上記ファイルを読み、`プロンプト本文（ここからコピー）` 以下を `prompt_text` として返却する

**UX設計方針：**
- 待ち時間が30秒以上になることを前提に、`待機UIはモーダルにしない`
- 解析中でも部品登録フォームの他項目は触れるようにする
- 自動解析が失敗しても、同じ画面のまま `貼り付け fallback` へ即切り替えられるようにする
- 複数PDFがある場合も、解析対象は1件だけ明示選択させて迷わせない

**データシート欄 UI 詳細：**

| 領域 | 役割 | UX上の意図 |
|---|---|---|
| PDFリスト | 選択済み PDF の一覧。各行に `解析対象` ラベルまたは `解析対象にする` 操作を置く | 複数添付時に、どのPDFを ChatGPT に送るかをユーザーが一目で判断できるようにする |
| アクションバー | 主ボタン `ChatGPTで解析`、副ボタン `ChatGPTから貼り付け`、任意ボタン `Geminiで解析` | 普段使いの主導線を左端に固定し、fallback は常に見える位置へ置く |
| 状態チップ | `Tampermonkey接続済み` `自動解析準備中` `ChatGPT待ち` `結果受信` | 「今どこで止まっているか」を短文で把握できるようにする |
| 解析ジョブカード | 4段ステップ `PDF準備` `ChatGPT起動` `解析待ち` `結果確認` を表示 | 進行状況が見えず再クリックを連打する事故を防ぐ |
| 解析結果レビュー | `helperResult` をレビュー用モーダルで表示し、内容修正・追加・削除を許可する | 候補が多い時も一覧性を保ち、未一致項目をその場で既存マスタへ紐付けられるようにする |

**ボタンの有効条件：**
- `ChatGPTで解析` は解析対象PDFが1件決まっている時だけ有効
- `ChatGPTで解析` は `window.__bitskeepTampermonkeyHelper` が検出できない場合、無効化せず案内モーダルを開く
- `ChatGPTから貼り付け` は常時使える fallback とする
- `Geminiで解析` は Gemini 設定済みの場合のみ表示する。主ボタンにはしない

**Tampermonkey + ChatGPT Web 自動解析フロー：**
1. ユーザーが部品登録画面で PDF を選び、`ChatGPTで解析` を押す
2. フロントが解析対象PDFを `POST /api/component-helper/chatgpt-jobs` へ送信する
3. サーバーは PDF を `storage/app/private/component-helper-temp/{uuid}.pdf` へ一時保存し、`temp_upload_token` `signed_download_url` `prompt_text` `expires_at` `sha256` を返す
4. BitsKeep 側 userscript が `GM_setValue` で job を公開し、`GM_openInTab` で ChatGPT タブを新規作成または再利用する
5. ChatGPT 側 userscript が job を検知し、`signed_download_url` から PDF を `Blob/File` として取得する
6. ChatGPT 側 userscript が `prompt_text` を入力し、PDF を添付して送信する
7. 応答が安定したら ChatGPT 側 userscript が JSON 本文または code block 内 JSON を抽出し、`GM_setValue` で結果を返す
8. BitsKeep 側 userscript が結果を受信し、既存の `parseChatGPTResult()` 経路へ流し込んでレビューパネルを表示する
9. ユーザーがレビューで適用を確定した後、部品保存時に `temp_datasheet_tokens[]` を一緒に送る
10. サーバーが temp PDF を正式な datasheet 保存先へ移動し、`component_datasheets` へ確定登録する

**なぜ結果 JSON を ChatGPT タブからサーバーへ直接 POST しないか：**
- `chatgpt.com` 側からの認証付き POST は CSRF / セッション / CORS の扱いが増え、単独運用の割に面倒が増える
- 今回の最小工数案では、**サーバーは temp PDF の保管と最終保存だけ** を担当し、解析結果は BitsKeep タブでレビューしてから通常の保存処理へ乗せる
- これにより、失敗時も `手動貼り付け` と同じ UI へ即座に切り替えられる

**一時PDFの扱い：**
- temp PDF の有効期限は 2 時間
- `signed_download_url` は自己完結型トークンとし、ChatGPT タブから cookie 無しで取得できる形式にする
- temp PDF は `保存成功` 時に正式保存先へ移動し、`消費済み token` は再利用不可
- 使われなかった temp PDF は期限切れ cleanup で削除する
- ユーザーが PDF をフォームから外した場合、対応する `temp_upload_token` もフォーム状態から外す

**レビューと適用のルール：**
- 自動解析と手動貼り付けの結果は、どちらもレビュー用モーダルへ必ず表示する
- レビュー対象は `part_number` `manufacturer` `common_name` `description` `分類候補` `パッケージ候補` `specs` とする
- レビュー中は `分類候補 / スペック候補` の追加・削除を許可する
- `分類候補` は複数候補を並べ、既存 `categories` へその場で紐付けた上で複数選択できる
- `パッケージ候補` は複数候補を並べ、既存 `package_groups / packages` へその場で紐付けた上で **登録対象は1件だけ選択** できる
- `specs` は `spec_types` へその場で手動紐付けできる
- 適用時に紐付け未完了の候補はスキップし、トーストで件数を返す
- 適用後も保存前なら値を書き換えられる。自動解析結果をロックしない

**異常系と fallback：**

| ケース | 画面の挙動 | ユーザーの次操作 |
|---|---|---|
| Tampermonkey 未接続 | 案内モーダルを表示し、`手動貼り付けへ切り替える` を出す | 貼り付けモーダルへ遷移 |
| ChatGPT 未ログイン | 状態チップを `ChatGPTログイン待ち` に変更し、ChatGPT タブを前面化 | ログイン後に `再開` |
| ChatGPT DOM 変更で自動化失敗 | 状態カードに `自動取得失敗` を表示し、`貼り付け fallback` を最前面に出す | ChatGPT の結果を手動貼り付け |
| JSON 抽出失敗 | raw テキストの冒頭を表示し、`貼り付けモーダルへコピー` ボタンを出す | ユーザーが不要部分を消して貼り付け |
| temp PDF 期限切れ | 状態カードを `再アップロード必要` にする | `ChatGPTで解析` を再実行 |
| ユーザーが途中キャンセル | 解析ジョブをローカル終了し、temp PDF は期限切れ cleanup に任せる | 再実行または手動入力 |
| 解析結果が空 | `データシートから情報を抽出できませんでした` を表示 | 手動貼り付け or 手入力 |

**Tampermonkey userscript の責務分離：**
- BitsKeep origin script
  - `window.__bitskeepTampermonkeyHelper` を公開して接続確認に使う
  - job 作成後の `GM_setValue` / `GM_addValueChangeListener` を担当する
  - JSON 結果を既存 Vue ロジックへ渡す
- ChatGPT origin script
  - job 取得、PDF download、prompt 入力、送信、結果抽出を担当する
  - DOM 変更や未ログイン時は失敗理由を job status として返す
  - **応答待機のストール判定**: 停止ボタン残留 + 90秒変化なし + 300文字以上 → 完了扱い
    - 思考モデル（o1/o3 系）は1分超の思考フェーズ中に短いテキストしか表示されないため、30秒・文字数無制限の旧判定では思考完了前に打ち切られていた
    - 300文字未満の短い応答はストール判定せず3分タイムアウトまで待機する

**Gemini 直結モードの位置づけ：**
- 既存の `POST /api/component-helper/analyze-datasheet` は残す
- ただし UI の主導線にはせず、連携設定で API キーを持つ環境だけの任意モードとする

### FNC-004 profile対応スペック検索

FNC-003 の一部。`component_specs.value_profile / value_numeric_typ / value_numeric_min / value_numeric_max` を使用する。  
`typ` と `保証境界(min/max)` は意味が違うため、一律の `BETWEEN` や一律の `overlap` に潰さない。

**照合基準ごとの判定：**

| 照合基準 | 対象 profile | 判定 |
|---|---|---|
| `typ` | `typ`, `triple` | `value_numeric_typ BETWEEN query_min AND query_max` |
| `range` | `range`, `triple` | `value_numeric_max >= query_min AND value_numeric_min <= query_max` |
| `max` | `max_only`, `triple` | `value_numeric_max BETWEEN query_min AND query_max` |
| `min` | `min_only`, `triple` | `value_numeric_min BETWEEN query_min AND query_max` |

検索条件の `spec_min / spec_max / spec_unit` もサーバー側で基底単位へ正規化してから比較する。  
そのため、以下は同じ意味として扱う。

- `1` + `uA`
- `0.000001` + `A`
- `1e-6` + `A`
- `1uA` + 単位空欄
- `1u` + `A`
- `1KΩ` + 単位空欄
- `1kΩ` + 単位空欄
- `1000` + `Ω`

検索は単位文字列の一致ではなく、正規化後の `query_min/query_max + normalized_unit` で比較する。したがって `1KΩ` と `1kΩ` はどちらも `1000 Ω` として扱う。単位だけを指定して数値範囲を指定しない検索は補助的な文字列検索に留め、パラメータ検索の主経路は必ず数値正規化を通す。

`max_only` を `typ` 検索へ混ぜたり、`typ` を `max` 検索へ混ぜたりしない。  
意味の違う値を曖昧に合流させると、設計上の余裕判断を誤らせるためである。

**パラメータ検索UI方針（2軸構造）：**

```
[種別コンボボックス              ▼]  [typ] [最大] [最小] [範囲] [3値]
 テキスト入力で候補をリアルタイム絞り込み   ↑ profileボタン群（未選択=全profile）
 → 候補から1件確定選択（自由文字列ではない）
```

- 種別はコンボボックス。入力中にリアルタイムで `name_ja / name_en / symbol / alias` を横断絞り込みし、最終的にドロップダウン選択で確定する（自由文字列のまま保存しない）
- profileはボタン選択。複数選択可。未選択時はprofile問わず全件を対象とする
- 2軸を独立操作することで「最大ベース電流で壊れないトランジスタを探す」（種別=ベース電流、profile=最大）と「typベース電流で互換品を探す」（種別=ベース電流、profile=typ）を自然に表現できる

### FNC-005 類似部品検索

**起点：** SCR-002 部品詳細画面の「類似部品を探す」ボタン

**ロジック：** 起点部品の以下の条件をパラメータとして SCR-001 に渡す
1. `category_id`（同分類）
2. 主要 `spec_type`（value / voltage / current / power）の `typ` 値 `value_numeric_typ` を起点に ±30% 範囲

`range / max_only / min_only` は類似度の代表値に使わない。  
理由は、保証境界や上限値を `typ` の代理にすると、実使用時の近さではなく「最悪条件の近さ」で候補が寄ってしまい、設計者の判断を誤らせるからである。

`triple` は `value_numeric_typ` を持つため、類似部品探索に利用できる。  
`typ` を持たない部品は、類似部品探索ボタンの表示条件から除外する。

**遷移先：** SCR-001（フィルター適用済み状態で表示）
**比較への導線：** SCR-001でヒット件数が複数の場合、チェックして SCR-004 へ進める

**「類似部品を探す」ボタン表示条件：** component_specs に主要スペックの `typ` または `triple.typ` が1件以上登録されている場合のみ表示

### FNC-024 Notion案件同期

**用途：** Notion 上の事業ページ群から対象案件DBを自動発見し、BitsKeep の `projects` テーブルへ同期して案件選択UIで統合的に扱えるようにする

- DB名が `01_案件管理`
- 親ページ名が `010_`〜`099_` に一致する事業
- 例: `010_綾電クリエイト / 01_案件管理`

- 対象DB/DataSource ID → `projects.source_database_key`

1. Notion 検索で `01_案件管理` を検索
2. database のみ抽出
3. ancestor/parent page 名が `^0[1-9][0-9]_.+` に一致するものだけ採用

5. 各DBの案件行を取得して upsert

- Notion側でアーカイブされた案件は `deleted_at` をセットせず、`sync_state='archived'` 相当で選択不可にする
- 同期失敗した事業DBは `sync_state='error'` とし、他事業の同期は継続する

### ⑪ 詳細ドラフト: 起動・停止 / リセット / 依存関係診断

**参照プロト：**
- `詳細仕様/UI構成/startup_sequence_diag.html`

**位置づけ：**
- `⑨ 斬新候補` および `⑩ 反証許可付き再評価` で最も強く押し上げられた候補
- 詳細ドラフト先行で具体化し、テンプレート限定の初版実装対象として扱う
- 実装進捗チェックリストへ反映する

**目的：**
- 複数電源、EN/RESET、PG、クロック、ブート条件、外部機器、バックパワー経路の依存関係をまとめて扱い、`本当に立ち上がるか` `安全に停止できるか` を判定する
- 電源タイミングを 1 本計算するのではなく、`安全な順序` `禁止順序` `危険な時間窓` `最小修正案` を返す

**想定する症状起点：**
- `起動しない`
- `再起動だけ失敗する`
- `電源OFF中に逆給電する`
- `停止時にハングする`

**入力項目：**

| 区分 | 項目 | 内容 |
|---|---|---|
| シナリオ | `scenario_type` | `cold_start` `warm_reset` `power_down` `hot_plug` `partial_power` |
| 電源レール | `rail_id` `label` | 例: `vin`, `3v3_main`, `1v0_core`, `2v5_io` |
| 電源レール | `start_delay_ms` `rise_time_ms` `fall_time_ms` | 立上り/立下りの相対時間 |
| 電源レール | `valid_min_v` `valid_max_v` | レールが `有効` と見なせる範囲 |
| 電源レール | `pg_assert_v` `pg_delay_ms` | Power Good の成立条件 |
| 制御信号 | `signal_id` `signal_type` | `EN` `RESET` `PG` `CLK_OK` `BOOT_OK` `DONE` など |
| 制御信号 | `source_node` `target_node` | どこがどこを制御するか |
| 制御信号 | `active_level` `release_delay_ms` | 極性と解放遅延 |
| ブロック | `block_id` `block_type` | `PMIC` `REGULATOR` `MCU` `FPGA` `ADC` `SENSOR` `EXT_DEVICE` |
| ブロック | `required_rails` `required_signals` | 成立に必要なレール/信号 |
| ブロック | `min_hold_ms` | リセット保持や電源安定待ちの最小時間 |
| 依存関係 | `from_node` `to_node` `condition_type` | `rail_valid` `pg_high` `reset_release` `clock_stable` `boot_media_ready` |
| バックパワー | `path_id` `from_domain` `to_domain` | 逆給電経路 |
| バックパワー | `via_type` | `ESD_DIODE` `PULLUP` `LEVEL_SHIFTER` `IO_BUFFER` `PROTECTION_PATH` |
| バックパワー | `series_resistance` `isolation_component` | 抑制手段の有無 |
| 外部条件 | `external_device_state` | 相手機器の電源状態、接続状態 |

**内部モデル：**
- ノード種別は `Rail` `Signal` `Block` の 3 種類とする
- 依存関係は有向グラフで表現し、各ノードは時間軸上で `off` `ramping` `valid` `released` `invalid` の状態を取る
- `Rail` は電圧レンジと PG 条件を持つ
- `Signal` は極性、源、伝播遅延、対象ノードを持つ
- `Block` は必要レール、必要信号、最小保持時間、禁止状態を持つ
- `BackpowerPath` は OFF ドメインへ流れ込む逆給電経路として別テーブルで評価する
- 評価器はシナリオごとに時系列シミュレーションし、成立順序と禁止順序を判定する

**判定ルール：**
1. `Block` は、必要な `Rail` がすべて `valid` かつ必要な `Signal` が成立するまで `RESET_RELEASED` にしてはならない
2. `PG` や `EN` が循環依存を作る場合は `起動デッドロック` と判定する
3. 対象ドメインが `off` のまま入力信号だけ High になる経路がある場合は `バックパワーリスク` と判定する
4. `Rail` が `falling` に入った後も外部 I/O や pull-up で給電される場合は `部分給電リスク` と判定する
5. `min_hold_ms` を満たさずに `RESET` が解放される場合は `再起動不安定` と判定する
6. 必要な `CLK_OK` `BOOT_OK` が未成立のまま起動順序が進む場合は `初期化順違反` と判定する
7. 条件不足で順序を確定できない場合は `判定保留` を返し、不足入力を列挙する

**出力：**
- 総合判定: `成立` `要注意` `不成立` `判定保留`
- `safe_start_sequence`: 安全な起動順序
- `safe_stop_sequence`: 安全な停止順序
- `forbidden_sequences`: 禁止順序一覧
- `danger_windows`: 危険な時間窓と理由
- `backpower_paths`: 検出した逆給電経路
- `first_suspects`: 最初に疑うべきレール/信号/依存関係
- `minimal_fixes`: 例 `RESETを120ms遅らせる` `series抵抗を追加` `EN条件をPG依存へ変更`
- `timeline_view_model`: UI でタイムライン描画するための正規化済みデータ

**暫定API案：**

| メソッド | パス | 内容 |
|---|---|---|
| POST | `/calc/startup-sequence/evaluate` | 起動/停止/依存関係診断を実行 |
| POST | `/calc/startup-sequence/templates` | よくある構成（PMIC+MCU, FPGA+DDR など）の雛形を生成 |

```json
  "forbidden_sequences": [
    {

    }
```

### ⑭ 詳細ドラフト: 熱設計 / 熱抵抗チェーン可視化

**参照プロト：**
- `詳細仕様/UI構成/thermal_chain_diag.html`

**位置づけ：**
- 既存の `熱見積もり / 温度上昇推定` 候補を、`抵抗 / Reg / Tr` 向けの実務ツールとして具体化したもの
- 詳細ドラフト先行で具体化し、代表パッケージ限定の初版実装対象として扱う
- 実装進捗チェックリストへ反映する

**目的：**
- 部品ごとの損失から `どこで何℃上がっているか` を熱抵抗チェーン単位で可視化する
- `Tj` だけを返すのではなく、`junction -> case/tab/body -> TIM -> sink/pattern -> ambient` の各段を積み上げて見せる
- 抵抗、レギュレータ、トランジスタで毎回バラバラにやっている熱設計の確認を、共通UIで短時間に終わらせる

**対象デバイス：**
- 抵抗器
- 線形レギュレータ / DC-DC の放熱評価
- トランジスタ / FET / 三端子Tr

**入力項目：**

| 区分 | 項目 | 内容 |
|---|---|---|
| デバイス | `device_type` | `resistor` `regulator` `transistor` |
| デバイス | `package_type` | 例: `1608`, `TO-220`, `DPAK`, `SOT-223` |
| 発熱 | `power_loss_w` | 平均損失 |
| 発熱 | `power_profile` | `steady` `pulse` `duty` |
| 環境 | `ambient_temp_c` | 周囲温度 |
| 定格 | `max_junction_temp_c` `max_case_temp_c` | 許容温度 |
| 熱経路 | `thermal_segments[]` | 各段の名称と熱抵抗値 |
| 熱経路 | `segment_type` | `junction-case` `case-tim` `tim-sink` `sink-ambient` `body-pad` `pad-ambient` |
| 実装条件 | `copper_area` `board_layers` | 基板放熱条件 |
| 放熱部材 | `heatsink_type` `interface_material` | ヒートシンク、絶縁シート、グリス |
| 比較条件 | `candidate_parts[]` | 複数部品比較時の候補 |

**内部モデル：**
- デバイスごとに `熱抵抗セグメント列` を持つ
- 1 セグメントは `name`, `rth_c_per_w`, `delta_t_c`, `cumulative_temp_c` を持つ
- 定常時は `delta_t = power_loss_w * rth_c_per_w` で各段を積み上げる
- パルス時は初版では `参考表示` とし、将来拡張として `Zth(t)` または `RC ラダー` を受けられる形にする
- 抵抗器は `body -> terminal -> pad -> ambient`、Reg/Tr は `junction -> case/tab -> TIM -> sink/board -> ambient` を基本テンプレートとする

**判定ルール：**
1. `cumulative_temp_c` が `max_junction_temp_c` を超える場合は `不成立`
2. `10℃未満` の温度マージンしか残らない場合は `要注意`
3. 最大寄与の熱抵抗セグメントを `支配段` として返す
4. ヒートシンク未指定で `sink-ambient` セグメントが必要な場合は `判定保留`
5. 候補比較時は `温度余裕`, `部品点数`, `放熱部材有無` で並べ替える

**出力：**
- 総合判定: `成立` `要注意` `不成立` `判定保留`
- `node_temps`: 各節点温度
- `segment_deltas`: 各熱抵抗での温度上昇
- `dominant_segments`: 支配的な熱抵抗段
- `required_rth_reduction`: 必要な熱抵抗削減量
- `candidate_comparison`: 候補部品/放熱案の比較
- `thermal_stack_graph`: 段積みグラフ描画用データ

**暫定API案：**

| メソッド | パス | 内容 |
|---|---|---|
| POST | `/calc/thermal-chain/evaluate` | 熱抵抗チェーン評価を実行 |
| POST | `/calc/thermal-chain/templates` | 抵抗 / Reg / Tr のテンプレートを返す |

**暫定レスポンス例：**
```json
{
  "status": "warn",
  "ambient_temp_c": 55,
  "node_temps": [
    { "name": "ambient", "temp_c": 55.0 },
    { "name": "sink", "temp_c": 66.2 },
    { "name": "case", "temp_c": 78.4 },
    { "name": "junction", "temp_c": 109.6 }
  ],
  "segment_deltas": [
    { "name": "sink-ambient", "rth_c_per_w": 5.6, "delta_t_c": 11.2 },
    { "name": "case-sink", "rth_c_per_w": 6.1, "delta_t_c": 12.2 },
    { "name": "junction-case", "rth_c_per_w": 15.6, "delta_t_c": 31.2 }
  ],
  "dominant_segments": ["junction-case"],
  "required_rth_reduction": 8.0
}
```

**UI方針：**
- 入口で `抵抗` `Reg` `Tr` を切り替える
- 上段に `総合判定` `最大温度` `次に削るべき熱抵抗段` を固定表示する
- 中央に `熱抵抗チェーンの段積みグラフ` を置き、各段の寄与がどこで効いているかを見せる
- 下段に `候補比較` `必要な放熱改善量` `支配段` を並べる
- `Tj だけ` ではなく `途中の各節点温度` を出し、構造関数風の読み方ができるようにする

### ⑮ 全体詳細設計【モジュール分割と責務境界】

BitsKeep 全体は、`在庫・案件の業務基盤` と `設計判断支援` を一つの画面群に見せるが、内部実装は責務ごとに分割する。特に、`高頻度・小修正系` と `高損害診断系` を同じ種類のツールとして扱わない。

### ⑰ 初版の実装順序と採用境界

**実装波次：**
1. 運用基盤を先に固める
   - ホーム、横断検索、案件コンボボックス、案件管理、監査ログ
   - この段階で `案件選択の迷い` と `部品運用` を止めないことを優先する
2. 計算系の初版本隊を入れる
   - エンジニア電卓
   - 抵抗/容量ネットワーク探索
   - ADC、誤差/歩留まり、電流検出、インタフェース余裕、電源余裕
3. 高頻度・小修正系を先に一部入れる
   - コネクタ視点補助
   - ケーブル/ジャンパ/0Ω系の軽量参照ツール
4. 高損害診断系の初版を限定投入する
   - 起動依存診断
   - 熱抵抗チェーン可視化
5. 入力保護/保護協調や観測性診断は、テンプレートとデータ整備の目処が立ってから投入する

**チェックリスト反映ルール：**
- 上記 1〜4 に入る機能だけを実装進捗チェックリストへ載せる
- 5 の群、および `⑯-2. 現時点では実現困難` に入る機能は、仕様書に残してもチェックリストには載せない

**採用してよい機能の条件：**
- 入力項目が整理でき、代表テンプレートで再現性のあるテストケースを作れる
- 出力が `次に何を直すか` `どの候補を選ぶか` に繋がる
- 仮定条件を明示できる
- 結果を `summary + warnings + chart_model + next_actions` に正規化できる

**まだ採用しない方がよい機能の条件：**
- 一般化しすぎると入力負荷だけ増え、初版の精度が下がる
- 設計ファイル、基板レイアウト、未整備の部品辞書など、まだ存在しない入力元に依存する
- 物理シミュレータ級の汎用性を目指さないと成立しない
- `面白い` が `毎月使う` に繋がらない

Laravel の Resource Controller 規約に従い、RESTful に設計する。すべて `/api/v1/` プレフィックス。認証は Laravel Sanctum（セッションベース）。

| メソッド | パス | 機能 | FNC |

| GET | `/components/{id}/similar` | 類似部品取得 | FNC-005 |
| GET | `/components/compare` | 比較データ取得（`?ids=1,2,3`） | FNC-006 |

`POST /components` と `PATCH /components/{id}` は、通常の `datasheets[]` に加えて `temp_datasheet_tokens[]` を受け付ける。`temp_datasheet_tokens[]` は `component-helper/chatgpt-jobs` で作成した一時PDFを正式 datasheet として確定するための token であり、保存成功時に消費済みになる。

補足:
- `GET /component-helper/temp-datasheets/{token}` は signed URL 用の Web ルートであり、API 一覧には含めない
