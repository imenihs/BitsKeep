# BitsKeep 部品在庫管理システム 仕様書

---

## 1. 目的（絶対にブレてはいけない軸）

> **電子回路設計者が「使うべき部品」を最短で判断できるようにすること**

これは単なる在庫管理ではない。

| NG 思想 | 正しい思想 |
|---|---|
| 在庫があるものを探す | 設計要件に適合する部品を見つける |
| 安いものを探す | 価格・入手性・実装性で最適を判断する |

---

## 2. ユーザー種別と権限

| 種別 | 権限 |
|---|---|
| 閲覧者 | 検索・閲覧・データ出力のみ |
| 編集者 | 入出庫・部品編集 |
| 管理者 | マスタ編集・ユーザー管理 |

---

## 3. 設計者の思考フロー（UI設計の基準）

UIはこの順序で情報を提示すること。**在庫は最後。**

1. スペック（電気的条件を満たすか）
2. 実装可否（パッケージ・サイズ）
3. データシート（設計確証）
4. 価格（コスト感覚での取捨選択）
5. 在庫（試作・手配の即時可否確認）

## 4. 機能一覧

### コア機能
- 簡易フリーワード検索（名称・型番・備考）
- フィルタ検索（分類・商社・在庫タイプ等、AND条件）
- 高度検索（メーカ・値範囲・温度特性・購入日・在庫数）
- 類似部品検索（同一機能・近似スペック）
- 既存部品への入庫・出庫処理
- 部品データ編集
- 新部品登録（既存部品情報の流し込み対応・CSVインポート対応）
- 部品情報削除（論理削除）
- プロジェクト部品価格リスト（商社選択可能）
- 棚卸機能
- 使用履歴（日時・出庫者）
- 在庫警告機能（部品ごとの在庫下限設定）
- ユーザー管理
- マスターテーブル編集
- プロジェクト管理（登録・編集・削除）

### 付加機能
- パッケージ一覧（画像・3Dモデル・PDF管理）
- 合成抵抗提案（分圧・抵抗比率・直列/並列合成、E系列近似算出）
- Altium Designer用ライブラリDB連携（専用テーブルへの追加・更新・削除）
- Leantimeからのプロジェクト情報インポート/エクスポート

## 6. データモデル

### 設計方針

| 管理内容 | 採用パターン |
|---|---|
| 多対多の関係 | 中間テーブル |
| 数値スペック（定格・寸法等） | `component_specs`（spec_typeで区別） |
| 不定型・自由入力項目 | `component_attributes`（キーバリュー） |
| 在庫の物理単位 | `inventory_blocks`（ロット・リール単位） |
| 入出庫履歴 | `transactions` |
| 操作監査 | `audit_logs` |

`components` は最小情報（型番・名称程度）で登録可能。段階的入力を前提とする。

### テーブル一覧

| テーブル名 | 用途 |
|---|---|
| `components` | 部品基本情報（型番・通称・在庫数・入手可否・ファイルパス等） |
| `component_specs` | スペック値（定格・寸法・電圧・電流等、単位付き） |
| `spec_types` | スペック種別と単位の定義マスタ（例: 電圧 + V/mV/kV） |
| `component_attributes` | 柔軟な属性追加項目（キーバリュー） |
| `attribute_keys` | キーバリュー属性で使うkey候補の定義 |
| `categories` / `component_category` | 分類（多対多） |
| `packages` / `component_package` | パッケージ情報（多対多・画像・寸法・PDF付き） |
| `projects` / `component_project` | 使用プロジェクト（構成部品数・使用数含む） |
| `locations` / `component_location` | 保管棚・QRコード対応の場所管理（階層構造あり） |
| `suppliers` | 商社マスタ |
| `component_suppliers` | 部品ごとの仕入情報（単価・商社型番・URL） |
| `supplier_price_breaks` | 数量条件ごとの単価定義（価格ブレーク） |
| `shipping_rules` | 商社ごとの送料条件（閾値・送料） |
| `inventory_blocks` | 在庫単位（パッケージ種別・ロット番号・リールID等） |
| `transactions` | 入出庫履歴（誰が・いつ・どれだけ・理由） |
| `component_altium_links` | Altium連携情報（sch/pcbライブラリパス・名称） |
| `users` | ユーザー情報（表示名・役割・認証情報） |
| `audit_logs` | 操作履歴ログ（変更差分をJSON形式で記録） |

### 主要テーブル定義

```sql
-- 部品基本情報
components (
  id, manufacturer, part_number, common_name,
  condition,             -- 新品/中古
  quantity_new, quantity_used,
  procurement_status,    -- EOL / Active 等
  datasheet_path, image_path,
  created_by (FK→users), updated_by (FK→users),
  deleted_at, created_at, updated_at
)

-- スペック値（定格・寸法等）
component_specs (
  id, component_id (FK), spec_type_id (FK), value
)

-- スペック種別と単位
spec_types (
  id, spec_name, unit, description
  -- 例: spec_name='voltage', unit='V'
  -- spec_typeによりunit候補が限定される
)

-- キーバリュー属性
component_attributes (
  id, component_id (FK), attribute_key_id (FK), value
)
attribute_keys ( id, name, code, description )

-- 仕入情報
component_suppliers (
  id, component_id (FK), supplier_id (FK),
  supplier_part_code, unit_price, product_url,
  created_at, updated_at
)
supplier_price_breaks (
  id, component_supplier_id (FK), min_quantity, unit_price
)
shipping_rules (
  id, supplier_id (FK), min_total_amount, shipping_fee
)

-- 在庫単位（ロット）
inventory_blocks (
  id, component_id (FK), component_supplier_id (FK),
  quantity, stock_type,  -- new/used
  reel_code, lot_number,
  location_id (FK), condition,
  created_by (FK), updated_by (FK),
  deleted_at, created_at, updated_at
)

-- 入出庫履歴
transactions (
  id, component_id (FK), inventory_block_id (FK),
  user_id (FK), type,  -- in/out
  quantity, reason, created_at
)

-- Altium連携
component_altium_links (
  id, component_id (FK),
  sch_lib_path, sch_lib_name,
  pcb_lib_path, pcb_lib_name,
  created_at, updated_at
)

-- 操作ログ
audit_logs (
  id, model_type, model_id, user_id,
  action, field_changes,  -- JSON形式
  created_at
)
```

## 9. UI設計ルール

### 全体共通
- ライト・ダークテーマ対応。カラーは `theme.css` の `var(--color-*)` トークンのみ使用。
- `null` 項目は表示しない。
- カード形式表示。背景は `card-even` / `card-odd` で交互塗り分け。

### 部品一覧画面
- 画面上部：フリーワード検索
- 画面左：フィルタ（分類・商社・在庫タイプ等）。フィルタが優先。
- 画面右上：テーマ切り替えボタン
- カード左：部品外観イメージ
- カード右（上から順）：名称 → 型番 → メーカ → 新品/中古在庫数 → データシートリンク（なければ非表示）→ パッケージ

### 部品詳細画面

**Aランク（設計初期に順に確認する情報）**

| 優先 | 種別 | 内容 |
|---|---|---|
| ① | 外観 | 画像サムネイル |
| ② | 基本情報 | 型番・通称・メーカ |
| ③ | スペック概要 | 値・定格電圧/電流/電力 |
| ④ | データシート | 主要リンク |
| ⑤ | 最安価格 | 商社名・価格（1件）・リンク |
| ⑥ | 入手可否 | 量産中/EOL/在庫限りなど |
| ⑦ | 在庫状況 | 新品/中古在庫数 |

**Bランク（詳細検討・社内管理用）**

| 種別 | 内容 |
|---|---|
| スペック詳細 | 許容差・温度特性・ランク・サイズ・パッケージ・色 |

## 10. 非機能要件

| 項目 | 方針 |
|---|---|
| 検索パフォーマンス | JOINの最適化必須 |
| 履歴 | transactionsは追記のみ（更新・削除しない） |
| 拡張性 | specはkey-value拡張、attributeで将来対応 |
| 整合性 | transactionsは必ずcomponent_idを持つ。在庫はinventory_blocksから算出可能 |
| ファイル管理 | 画像・データシートはDBに保存しない。指定フォルダにユニークファイル名で管理 |

## 13. DB設計修正（確定）

### spec_types / spec_units の分離

`spec_types` から `unit` を分離し、単位候補を `spec_units` テーブルで1対多管理する。

```sql
-- 修正前（誤）
spec_types ( id, spec_name, unit, description )

-- 修正後（正）
spec_types ( id, spec_name, description )
spec_units ( id, spec_type_id (FK), unit_label )
-- 例: spec_type_id=1(voltage) → V, mV
--     spec_type_id=2(resistance) → Ω, kΩ, MΩ
```

### component_specs に value_numeric 追加

数値範囲検索のために `value_numeric REAL` カラムを追加する。

```sql
component_specs (
  id,
  component_id (FK),
  spec_type_id (FK),
  value,          -- 表示用文字列（例: "10k"）
  value_numeric,  -- 検索用正規化数値（例: 10000.0）基準単位で保存
  unit            -- 選択された単位（例: "kΩ"）
)
```

**正規化ルール（登録時に value_numeric を算出）：**

| spec_type | 基準単位 | 変換例 |
|---|---|---|
| resistance | Ω | 10kΩ → 10000 |
| voltage | V | 100mV → 0.1 |
| current | A | 50mA → 0.05 |
| power | W | 500mW → 0.5 |
| frequency | Hz | 16MHz → 16000000 |
| dimension | mm | 1cm → 10 |

```
[左サイドバー w-64]  [右メインエリア flex-1]

```

### FNC-004 数値範囲検索

### FNC-012 出庫処理

**起点：** SCR-001 または SCR-002 の「出庫」ボタン（編集者以上）

**入力項目：**

| 項目 | 必須 | 内容 |
|---|---|---|
| 部品 | ✓ | 起点の部品（自動セット or 検索） |
| inventory_block | ✓ | どのロット/リールから出庫するか選択。在庫数・保管場所・ロット番号を表示して選ばせる |
| 数量 | ✓ | 整数・1以上・選択ブロックの在庫数以下 |
| プロジェクト | | projectsから選択（多対多） |
| 理由 | | テキスト（任意） |

**処理：**
1. `transactions`（type='out'）に記録
2. `inventory_blocks.quantity` を減算。0になった場合は `deleted_at` をセット（論理削除）
3. `components.quantity_new` または `quantity_used` を減算
4. プロジェクト指定がある場合は `component_project` に記録

**バリデーション：** 出庫数 > ブロック在庫数は不可。在庫0の部品への出庫は不可。

## 18. 設計決定事項（追記）

### ① 棚卸QRコードスキャン対応【決定: 対応する】

DB設計（`locations.qr_code`）はすでに対応済み。棚卸画面（FNC-013）にQRスキャン機能を実装する。

**実装方針：** ブラウザの `getUserMedia` API でカメラ起動、`jsQR` ライブラリでQRコードを読み取り対象棚を自動選択する。PCブラウザ・スマホブラウザ双方で動作する。

**棚卸フロー（QRスキャン版）：**
```
スマホで棚のQRコードをスキャン
  ↓
その棚の inventory_blocks 一覧が表示される
  ↓
実数を入力して確定（差分ハイライト付き）
```

---

### ② 在庫警告の通知方法【決定: 画面表示のみ（初期スコープ）】

- 通知はアプリ内の視覚表示のみとする
- SCR-001 一覧の警告アイコン・SCR-002 詳細の着色で十分
- メール/Push通知は運用開始後に実需が発生してから追加する（守りのフェーズの原則）

---

### ③ CSVインポートの列フォーマット【確定】

移行元データなし。以下のフォーマットで定義する。

**必須列：** `part_number`, `name`, `manufacturer`
**任意列：** `common_name`, `category`, `package`, `value`, `value_unit`, `voltage`, `current`, `power`, `tolerance`, `procurement_status`, `supplier_name`, `supplier_price`, `supplier_url`, `supplier_part_code`, `quantity_new`, `quantity_used`, `datasheet_url`, `notes`

- 1行目はヘッダ行
- 文字コード: UTF-8（BOM付き可）
- 日付形式: YYYY-MM-DD
- `procurement_status` の値: `active` / `eol` / `limited` / `recommended_stop`
- 同一 `part_number` が既存レコードと一致する場合は上書き更新
